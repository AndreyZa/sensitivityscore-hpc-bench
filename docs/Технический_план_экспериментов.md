# Технический план экспериментов: SensitivityScore vs K8s/KubeVirt/Slurm/Slinky на Geant4

> Рабочий документ для себя. Цель — довести гипотезы H1–H4 до конкретных артефактов: код, манифесты, скрипты сбора метрик, пайплайн анализа. Каждый раздел = что именно делать руками.

---

## 0. Структура работы по этапам

```
Этап 0: Инфраструктурная готовность      (Slurm/K8s версии, доступ к PMU)
Этап 1: Образ нагрузки Geant4 + профили S
Этап 2: Доработка SensitivityScore под эксперимент
Этап 3: Агент сбора метрик (perf_event_open → Redis)
Этап 4: Harness оркестрации экспериментов (запуск серий, overcommit, повторы)
Этап 5: Конфигурации A → B → C → D (по очереди)
Этап 6: Пайплайн анализа данных + проверка H1–H4
```

Ниже — техническая начинка каждого этапа.

## 1. Образ нагрузки: Geant4 с управляемым профилем чувствительности

### 1.1. Базовый образ

Собрать Docker-образ Geant4 (рекомендую на базе `geant4-data` + multistage build, чтобы не тащить toolchain в финальный слой):

```dockerfile
FROM cern/geant4:11.2 AS build
# physics lists, data sets уже внутри базового образа CERN, либо собрать из исходников
# если нужен контроль версии — собрать из исходников с -DGEANT4_INSTALL_DATA=ON

FROM debian:bookworm-slim
COPY --from=build /usr/local/geant4 /usr/local/geant4
ENV G4DATA=/usr/local/geant4/share/Geant4/data
COPY entrypoint.sh /entrypoint.sh
COPY macros/ /macros/
ENTRYPOINT ["/entrypoint.sh"]
```

Не обязательно писать собственное Geant4-приложение с нуля — для эталонной нагрузки подходит пример из `examples/extended/` (например, `electromagnetic/TestEm5` или `hadronic/Hadr03`), он уже даёт реалистичный профиль cross-section lookups.

### 1.2. Параметризация профиля S через переменные окружения

Это центральная техническая идея: **один и тот же бинарник, разный профиль интерференции**, управляемый через env/CLI:

| Параметр | Low-S | High-S | Влияет на |
|---|---|---|---|
| `G4_THREADS` | 1 | равно числу физических ядер на NUMA-домене | NUMA remote-access ratio |
| `PHYSICS_LIST` | `QGSP_BERT` (компактные таблицы) | `FTFP_BERT_HP` (большие таблицы сечений, HP = high precision neutron data) | LLC working set |
| `N_PRIMARIES` | 10⁴ | 10⁶–10⁷ | длительность job, стабильность измерения |
| `OUTPUT_MODE` | `none` / агрегат в конце | построчная запись каждого N-го события в ROOT/ntuple | Disk I/O чувствительность |
| `RNG_SEED` | фиксирован для каждого повтора | — | воспроизводимость |

```bash
# entrypoint.sh (упрощённо)
exec geant4-app \
  -t ${G4_THREADS:-1} \
  -p ${PHYSICS_LIST:-QGSP_BERT} \
  -n ${N_PRIMARIES:-10000} \
  -o ${OUTPUT_MODE:-none} \
  -m /macros/run.mac
```

**Действие:** реализовать `entrypoint.sh` + два готовых k8s-манифеста (`job-low-s.yaml`, `job-high-s.yaml`) с этими env уже выставленными — чтобы не собирать вручную каждый раз.

### 1.3. Аннотации для SensitivityScore

Job должен декларировать свой профиль явно (MVP сейчас на annotation-based флагах — используем тот же механизм, не выдумываем новый):

```yaml
metadata:
  annotations:
    scheduling.phd/sensitivity-llc: "high"      # high|medium|low
    scheduling.phd/sensitivity-numa: "high"
    scheduling.phd/sensitivity-net: "low"
    scheduling.phd/sensitivity-io: "low"
    scheduling.phd/job-id: "geant4-highs-rep03"  # для корреляции с метриками в Redis
```

## 2. Доработка плагина SensitivityScore

Текущий MVP: annotation-based флаги + JSON-кэш метрик узла. Для серии экспериментов нужно расширить под два момента.

### 2.1. Score function — формализовать прямо сейчас, пока не размылось

Раз диссертация формализует Z = {G, R, S}, score-функция в коде должна явно отражать скалярное произведение профиля job и состояния узла:

```go
func (p *SensitivityScorePlugin) Score(ctx context.Context, state *framework.CycleState,
    pod *v1.Pod, nodeName string) (int64, *framework.Status) {

    jobProfile := extractSensitivityVector(pod.Annotations) // S_job = (llc, numa, net, io) ∈ [0,1]^4
    nodeState := p.metricsCache.Get(nodeName)                // текущая нагрузка узла по тем же 4 измерениям

    // Чем выше произведение профиля job и текущей загруженности узла по тому же измерению —
    // тем больше риск интерференции → тем ниже score.
    interference := dotProduct(jobProfile, nodeState.Pressure)

    score := normalize(maxScore - interference*weight)
    return score, nil
}
```

**Действие:** вынести веса (`weight` по каждому измерению S) в ConfigMap — это пригодится в Главе 3 для абляционного исследования (ablation: что будет, если убрать NUMA-измерение из score).

### 2.2. Источник `nodeState.Pressure` для каждой конфигурации стенда

Это важно отдельно прописать, потому что источник метрик различается между A/B:

- **Конфигурация A (bare-metal под)**: `nodeState.Pressure` читается из Redis, который наполняется DaemonSet-агентом (см. §3) напрямую с PMU хоста.
- **Конфигурация B (KubeVirt)**: то же самое, но Pressure агрегируется по cgroup процесса `qemu-kvm`, а не по cgroup пода — нужен отдельный resolver, который мапит `VMI → qemu PID → cgroup path`. Реализовать как `NodeStateResolver` интерфейс с двумя реализациями (`PodCgroupResolver`, `QemuProcessResolver`), чтобы не плодить if-ветки в самом плагине.

```go
type NodeStateResolver interface {
    Resolve(nodeName string) (PressureVector, error)
}
```

## 3. Агент сбора метрик: perf_event_open() → Redis

Это тот компонент, который сейчас замещает JSON-кэш — фиксируем его архитектуру до начала измерений, чтобы не переписывать на полпути.

### 3.1. DaemonSet-агент (Go)

Один под на узел, с привилегией `CAP_PERFMON` (не полный `privileged: true` — это будет аргументом в защите, что подход безопаснее, чем грубый privileged-доступ):

```yaml
securityContext:
  capabilities:
    add: ["PERFMON", "SYS_PTRACE"]   # SYS_PTRACE нужен только если читаем /proc/<pid>/stat других неймспейсов
```

Счётчики, которые открываются через `perf_event_open()` на cgroup пода (`PERF_TYPE_HARDWARE`, cgroup-scoped events начиная с ядра 4.x):

| Счётчик | `perf_event_open` config | Измерение S |
|---|---|---|
| `PERF_COUNT_HW_CACHE_MISSES` (LL) | `PERF_TYPE_HW_CACHE \| LL \| MISS` | LLC |
| `PERF_COUNT_HW_CACHE_REFERENCES` | то же, READ/ACCESS | LLC (нормировка) |
| `node-load-misses` / `node-loads` (raw event, требует знания PMU модели CPU — Intel uncore counters) | raw config, через `/sys/bus/event_source/devices/uncore_imc_*` | NUMA bandwidth |
| Network bytes | проще брать не из perf, а из cgroup `net_cls`/eBPF socket hook отдельно | Network |
| Block I/O | cgroup `io.stat` (blkio v2) | Disk I/O |

**Важный технический момент**: LLC и NUMA — честные PMU-счётчики через `perf_event_open()`. Network и Disk I/O дешевле и надёжнее брать не из perf, а из стандартных cgroup v2 файлов (`io.stat`, `net` через nftables/eBPF), не нужно изобретать. Не смешивать источники в одной абстракции — но нормализовать в единый PressureVector перед записью в Redis.

### 3.2. Схема в Redis

```
HSET node:metrics:<node_name>           llc_miss_rate <f> numa_remote_ratio <f> net_bw <f> io_iops <f> ts <unix>
HSET job:metrics:<job_id>:<node_name>   llc_miss_rate <f> numa_remote_ratio <f> ... ts <unix>
EXPIRE node:metrics:<node_name> 30      # TTL, чтобы не читать протухшие данные планировщиком
```

Два набора ключей: `node:metrics:*` — для самого планировщика (текущее давление на узел, нужно для Score), `job:metrics:*` — для последующего анализа экспериментов (находится по `scheduling.phd/job-id` из аннотации, пишется агентом весь жизненный цикл job, не только на момент скоринга).

**Действие:** добавить `job:metrics:*` write path в агент — сейчас в MVP, скорее всего, есть только `node:metrics:*` (нужный для рантайм-скоринга), а под эксперимент нужна полная история по job для последующего расчёта makespan/дисперсии.

### 3.3. Конфигурация B (KubeVirt) — что реально измеримо

Зафиксировано в предыдущем документе для партнёров, повторю как техническое решение для агента: вместо чтения PMU из гостевой VM (требует vPMU passthrough, не факт что доступно на стенде) — агент резолвит `VMI → qemu-kvm PID` через `virt-launcher` под и читает PMU host-side по cgroup этого процесса. Написать health-check при старте серии B: если `vPMU` доступен (можно проверить наличием `/sys/devices/cpu/...` внутри VM или через `virsh capabilities`), переключаться на честный in-guest сбор; если нет — фиксировать факт приближения в метаданных прогона (`approximation: host-side`), чтобы потом не перепутать сопоставимость данных A и B.

## 4. Harness оркестрации экспериментов

Не повторять руками `kubectl apply` 10 раз на точку плана — отдельный скрипт-раннер (Python/Go, не принципиально, Python быстрее для итерации):

```python
# run_experiment.py
CONFIGS = ["A", "B", "C", "D"]
PROFILES = ["low-s", "high-s"]
OVERCOMMIT = [1.0, 1.5, 2.0]
REPETITIONS = 10

for config in CONFIGS:
    for profile in PROFILES:
        for ratio in OVERCOMMIT:
            for rep in range(REPETITIONS):
                job_id = f"{config}-{profile}-oc{ratio}-rep{rep}"
                submit_job(config, profile, ratio, job_id)   # kubectl apply / sbatch, в зависимости от config
                wait_for_completion(job_id)
                record_result(job_id)                         # makespan + выгрузка job:metrics:* из Redis в Parquet
                cooldown(seconds=30)                           # дать узлу остыть от предыдущего прогона
```

Технические детали по веткам `submit_job`:

- **A / B (K8s/KubeVirt)** — `kubectl apply -f` сгенерированного из шаблона манифеста (Jinja2-шаблон с подстановкой profile/overcommit/job_id), дождаться `Succeeded` через `kubectl wait`.
- **C (Slurm)** — `sbatch` со скриптом, аналогичные env-параметры передаются через `--export=`, ожидание через `squeue`/`sacct`, makespan берётся из `sacct -j <id> --format=Elapsed`.
- **D (Slinky/slurm-bridge)** — те же K8s-манифесты, что для A, но под транслируется в Slurm job через slurm-bridge — makespan мерить и со стороны K8s (`kubectl get pod`), и со стороны `sacct`, чтобы видеть оверхед самой трансляции (это отдельная метрика, не в H1–H4, но полезно для обсуждения с партнёрами).

**Overcommit ratio технически** — это не нативная настройка Geant4, а способ упаковки job на узел: при ratio 1.5 на узел с capacity на 4 параллельных job сабмитим 6, полагаясь на то, что планировщик (default или SensitivityScore) сам разрешает конфликт размещения. Для C (чистый Slurm) это управляется через `--oversubscribe` в `sbatch`/partition config.

## 5. Формат данных и пайплайн анализа

### 5.1. Схема результатов (Parquet/CSV)

```
config | profile | overcommit | rep | node | makespan_s | llc_miss_rate | numa_remote_ratio | net_bw | io_iops | approximation
```

### 5.2. Статистический анализ

- Между конфигурациями на одной точке плана (одинаковые profile/overcommit) — **Mann-Whitney U** (не ANOVA/t-test по умолчанию: 10 повторов — маленькая выборка, нет оснований предполагать нормальность HPC-таймингов, у них часто правый хвост из-за шумных соседей).
- Эффект размера — **Cliff's delta** (устойчивее к выбросам, чем Cohen's d, для таких данных).
- Дисперсия (для проверки "стабильности" в H1) — сравнение **coefficient of variation (CV = σ/μ)** между A(default) и A(SensitivityScore), не просто σ — иначе разные по абсолютному makespan конфигурации некорректно сравнивать по дисперсии напрямую.

### 5.3. Визуализация

- Boxplot makespan по `config × profile` при фиксированном overcommit=2.0 (где интерференция максимальна — это рабочая точка, где должны разойтись A(default) и A(SensitivityScore)).
- Scatter LLC miss rate vs makespan, отдельно для high-S/low-S — проверка, действительно ли LLC давление коррелирует с деградацией (обосновывает выбор LLC как измерения S, а не просто декларативно).
