# Глава 3. Экспериментальное исследование метода интерференционно-чувствительного планирования (черновик)

> Черновик текста главы; нумерация секций условная. Числа — из серий STAGE
> (июль 2026), воспроизводимы из ClickHouse (`make ch-analyze STAND=stage
> RUN_LABEL=<серия>`). Пометки [ОЖИДАЕТСЯ] — места под результаты
> проверочной серии исправленного скоринга. Конфигурации B (KubeVirt),
> C (Slurm), D (Slinky) в главу войдут после прод-стенда — здесь каркас
> только по конфигурации A.

## 3.1. Цели, задачи и гипотезы экспериментального исследования

Целью экспериментального исследования является количественная проверка
основного положения работы: планирование, учитывающее вектор
чувствительности задачи S = (LLC, NUMA, Net, IO) ∈ [0,1]⁴ из модели
Z = {G, R, S} (Глава 2), даёт измеримое преимущество перед планировщиками,
не имеющими модели интерференции, в условиях совместного размещения
разнородных задач.

Проверяемые гипотезы (полные формулировки — Программа экспериментов, §6):

- **H1**: планировщик SensitivityScore обеспечивает меньшее время
  выполнения и меньшую дисперсию по сравнению со штатным kube-scheduler
  при совместном размещении задач с различными профилями чувствительности.
- **H1-t** (контрольная): преимущество обеспечивается именно вектором
  чувствительности, а не учётом загрузки узлов как таковым — сравнение
  с load-aware планировщиком Trimaran (`LoadVariationRiskBalancing`),
  учитывающим утилизацию CPU/памяти, но не имеющим модели интерференции.
- **H2–H4**: сохранение эффекта в виртуализированной среде (KubeVirt),
  сравнение со Slurm и гибридной схемой Slinky — вне данной серии,
  требуют конфигураций B/C/D.

Задачи исследования: (1) построить измерительную инфраструктуру,
снимающую давление узлов по всем осям S без участия испытуемых
планировщиков; (2) построить методику, разделяющую *качество решения*
планировщика и *исход* (время выполнения), подверженный шуму; (3) провести
серии с фоновой нагрузкой по каждой оси и со смешанной нагрузкой;
(4) выявить и устранить систематические ошибки измерений; (5) по
результатам — уточнить модель скоринга и проверить уточнение отдельной
серией с заранее объявленными критериями.

## 3.2. Экспериментальный стенд

### 3.2.1. Аппаратная и программная конфигурация

Облачный стенд STAGE: управляемый кластер Kubernetes (k0s v1.35.6,
Timeweb Cloud), измерительные узлы 2 vCPU / 1.9 GiB (Ubuntu 24.04,
cgroup v2), управляющий слой вынесен провайдером. Системные компоненты
эксперимента (Redis, испытуемый планировщик, metrics-server) изолированы
на выделенном узле ролью `node-role.kubernetes.io/ss-system` с taint —
измерительные узлы несут только задачи и генераторы нагрузки, фонового
загрязнения метрик системными компонентами нет.

Две конфигурации измерительного пула: серии «диск» и «сеть» — 6 узлов
(включая узел w7, аппаратно ~2× медленнее остальных — выявлено пиннед-
пробами, см. 3.3.4); серии «кэш» и «смешанная» — 3 узла с подтверждённым
доступом к PMU (эталонные времена однородны: 68–74 с, ±2–3 %).

Виртуализация KVM данного провайдера честно виртуализирует cgroup-scoped
PMU (проверено диагностикой perfcheck: ~3·10⁶ cache-misses/с на пробной
нагрузке); встречены гипервизоры, где то же открытие счётчика проходит,
но чтение возвращает ноль (VMware Workstation) — поэтому проверка PMU
включена в процедуру ввода любого нового стенда (Этап 0).

### 3.2.2. Эталонная нагрузка

Эталонная задача — Geant4 11.2 (Монте-Карло симуляция прохождения частиц
через вещество; класс Particle Methods по таксономии «13 Dwarfs»). Один
образ, профиль чувствительности управляется параметрами:

| Профиль | Параметры | Заявленный S |
|---|---|---|
| low-s | 1 поток, без вывода | низкий по всем осям |
| high-s | MT, полный physics list | LLC, NUMA — high |
| high-s-io | + fsync-запись результатов порциями | LLC, NUMA, IO — high |
| high-s-net | помечен чувствительным к сети¹ | Net — high |

На 2-vCPU узлах STAGE профили масштабированы (2 потока, CPU-квота 500m,
3·10⁵ первичных частиц; время изолированного выполнения ~70 с — выше
разрешающей способности измерений, ниже порога утомительности серий).

¹ Профиль не генерирует собственный сетевой трафик (осознанное
ограничение, см. 3.6): сценарий сети проверяет обнаружение давления и
качество размещения, но не деградацию собственного обмена задачи.

### 3.2.3. Измерительная инфраструктура

Агент (DaemonSet, Go) каждые 5 с измеряет давление узла по осям S
независимо от источника нагрузки и пишет в Redis (ключи `node:metrics:*`
с TTL — горячий путь планировщика; `job:metrics:*` — накопительно для
анализа). Датчики по осям:

- **LLC** — промахи последнего уровня кэша в секунду, cgroup-scoped
  `perf_event_open` (пары счётчиков в perf-группе — числитель и
  знаменатель планируются на PMU совместно), нормировка на калибровку
  стенда `LLC_REFERENCE_MISSES_PER_SEC`;
- **NUMA** — доля удалённых обращений (`node-load-misses/node-loads`);
  на одноузловых (по NUMA) машинах STAGE ось отключена честно, нулевым
  весом;
- **IO** — PSI-доля времени ожидания диска (cgroup v2 `io.pressure`,
  нормирована ядром в [0,1]); сырые IOPS — только для анализа;
- **Net** — суммарный трафик подов к эмпирически калиброванной полосе
  стенда (`NET_REFERENCE_MBPS`, измеряется iperf3 между узлами через
  фактический CNI-путь; на STAGE — 1616 Мбит/с).

Существенный отрицательный результат, повлиявший на конструкцию датчика
LLC: классическое *отношение* промахов к обращениям под потоковой
нагрузкой инвертируется — знаменатель (обращения) растёт быстрее
числителя, и загруженный узел выглядит «чище» простаивающего (измерено:
0.018 на узле с потоковым генератором против 0.33–0.41 на простаивающих).
Планировщик, получающий такое отношение, систематически предпочитал бы
загруженный узел. Датчик переведён на промахи/сек с пер-стендовой
калибровкой (разделение простой/шторм на STAGE: 1.0–1.3·10⁶ против
15.2·10⁶ промахов/с, контраст ×12–15).

### 3.2.4. Генераторы фоновой нагрузки

По одному типу на ось: диск — `stress-ng --hdd` (непрерывная запись),
кэш — `stress-ng --stream` (потоковый проход по памяти), сеть — пары
iperf3 (UDP, 2×400 Мбит/с ≈ насыщение калиброванной полосы). Генераторы
объявляют минимальные ресурсные требования — для планировщиков без модели
интерференции загруженный узел выглядит наиболее свободным (конструкция
«ловушки»). Показатели генераторов не измеряются и в результаты не входят.

## 3.3. Методика

### 3.3.1. Сравниваемые планировщики

`default` (штатный kube-scheduler), `SensitivityScore` (исследуемый),
`trimaran` (`LoadVariationRiskBalancing` — контроль «достаточно ли любого
учёта загрузки»). Все три работают в одном кластере одновременно (выбор —
`schedulerName` задачи), генераторы и поток задач для всех плеч идентичны.

### 3.3.2. Метрики

Разделены *метрики решения* (какой узел выбран) и *метрики исхода*
(что получилось по времени):

- **ошибка размещения** (placement regret) — превышение интерференции
  выбранного узла над лучшим доступным на момент решения, по снапшоту
  давления из Redis; вычисляется оркестратором единообразно для всех
  плеч (в плече default плагин не участвует в размещении);
- **попадание на перегруженный/токсичный узел** — дискретная метрика
  решения (для смешанного сценария — на узел «своей» оси);
- **время выполнения** (makespan) — от кластерных временных меток
  контейнера (исключает очередь и загрузку образа);
- **замедление** — makespan к медиане изолированного эталона *того же
  профиля на том же узле* (см. 3.3.4);
- **стабильность** — коэффициент вариации времени между повторениями.

Метрики решения почти детерминированы и несут механизм; метрики исхода
шумны, но являются конечной целью. Расхождение между ними — само по себе
результат (см. 3.4.3–3.4.5).

### 3.3.3. Дизайн серий

Каждая серия: 3 плеча × 10 повторений × 6 задач-жертв (180 наблюдений).
Внутри повторения задачи поступают пуассоновским потоком (средний интервал
25 с); моменты прибытия детерминированы сидом номера повторения — все
плечи и повторные серии получают идентичные потоки (парный дизайн).
Два типа сценариев:

- *одноосевые* (диск, сеть, кэш): генераторы одной оси на одном узле,
  остальные чистые; выигрышная стратегия — уклонение;
- *смешанный*: на каждом узле генератор своей оси (кэш/диск/сеть), чистых
  узлов нет, поток — смесь трёх профилей (2+2+2, чередование); выигрышная
  стратегия — сопоставление профиля задачи и оси узла; уклонение «от
  нагрузки вообще» не работает по построению.

Статистический аппарат: агрегация до уровня повторения (члены одного
повторения интерферируют друг с другом и не независимы), критерий
Манна-Уитни, поправка Холма в семье (сценарий, метрика), размер эффекта
Cliff's δ.

### 3.3.4. Методологические правила, установленные в ходе работы

Три правила, каждое из которых устраняет обнаруженную систематическую
ошибку; предлагаются как часть методики для любых облачных стендов:

1. **Пер-узловые эталоны.** «Одинаковые» облачные узлы бывают аппаратно
   разными (узел w7 — 141 с против 71–74 с у соседей на идентичной
   задаче, ~2×; обнаружено пиннед-пробами после пересоздания узла).
   Замедление к общему эталону смешивает эффект планирования с
   неоднородностью железа; знаменатель обязан быть парой (профиль, узел).
2. **Эталоны в ту же сессию, что серия.** Производительность облачных
   vCPU дрейфует в течение суток; измеренный сдвиг между ночными
   эталонами и дневной серией +13–23 % — больше большинства исследуемых
   эффектов. Метрики решения от дрейфа не зависят, метрики исхода —
   критически.
3. **Проверка датчиков штормом до серии** (smoke): инверсия LLC-отношения
   (3.2.3) была поймана именно предсерийной проверкой, а не после
   недельной серии.

## 3.4. Результаты

### 3.4.1. Верификация профилей (fingerprint)

Заявленные векторы S сверяются с измеренными в изолированных эталонных
прогонах; требование монотонности (заявленный high измеряется выше
заявленного low по своей оси) выполняется по осям LLC и IO; по оси Net —
заявленное high при измеренном ~0 (профиль не генерирует трафик) —
зафиксировано как ограничение (3.6).

### 3.4.2. Одноосевые серии: диск и сеть

Ошибка размещения (среднее): диск — SensitivityScore 0.001 против 0.113
(default) и 0.166 (trimaran), p_holm ≈ 4·10⁻⁴, δ = −0.96…−1.0; сеть —
0.000 против 0.083 и 0.050. По времени выполнения на дисковой оси —
полное разделение выборок (δ = −1.0 против обоих сравнений). Load-aware
контроль (H1-t) на этих сериях не лучше default: агрессоры с минимальными
requests невидимы и для ресурсной модели, и для CPU-утилизации.

### 3.4.3. Одноосевая серия: кэш — первое расхождение метрик

По метрике решения SensitivityScore выигрывает (попаданий в шторм 3/60
против 20/60 у default; ошибка размещения 0.274 против 0.473,
p_holm = 0.003, δ = −0.84), но по времени — проигрывает default
(86.7 против 82.7 с, p = 0.02, δ = +0.69). Причина: стоимость
*скученности* — шесть кэш-чувствительных задач, уведённых с загруженного
узла, взаимно интерферируют на двух чистых узлах — превысила стоимость
самого кэш-шторма, оказавшегося при CPU-квотах 500m почти бесплатным.

### 3.4.4. Смешанная серия: сопоставление вместо уклонения

По метрике решения — максимальный результат: SensitivityScore ни разу из
60 не поставил задачу на генератор её собственной оси (default — 21/60,
trimaran — 22/60; trimaran свёз 35/60 задач на дисковый узел, «свободный»
по CPU-метрике, в том числе 10 из 20 диск-чувствительных — систематическая
ошибка одномерной метрики). По времени: диск-профиль значимо лучше
trimaran (83.2 против 91.0 с, p_holm = 0.002, δ = −0.94), сеть-профиль
направленно лучше обоих, но кэш-профиль значимо хуже default (91.3 против
78.7 с, δ = +0.95).

Эмпирическая матрица токсичности (время профиль × узел по всем плечам)
объясняет оба расхождения: дисковый генератор токсичен для **всех**
профилей (+15–18 с любому: iowait и вытеснение страничного кэша действуют
на узел целиком), кэш-генератор почти безвреден (узел с ним — быстрейший
даже для кэш-чувствительных задач). Планировщик, оптимизирующий формальную
метрику без знания *цены* оси, уводит кэш-задачи с эмпирически лучшего
узла на эмпирически худший.

### 3.4.5. Калибровка цен осей

По 360 наблюдениям двух серий оценена модель
`замедление − 1 ≈ γ·N_соседей + Σ_a p_a·(α_a + β_a·s_a)` (неотрицательный
МНК, посерийная константа поглощает сессионный дрейф; R² по ячейкам
профиль × узел = 0.88):

| ось | α (базовая цена, платят все) | β (надбавка чувствительной) |
|---|---|---|
| диск | **0.24** | 0 |
| кэш | 0 | 0 |
| сеть | 0.02 | 0 |

γ = 0.026 на соседа; сессионный дрейф +0.13…+0.23 — крупнейший член.

Два вывода уровня модели. Первый: цена оси может быть **базовой** —
давление оси замедляет любую задачу узла независимо от её заявленной
чувствительности; скоринг вида Σ w·s·p принципиально не способен её
учесть (задача с s = 0 не платит ничего при любых весах) — перевзвешивание
осей недостаточно. Второй: на стенде с CPU-квотами пер-задачное
различение (β > 0) не проявляется — все профили фактически compute-bound;
проверка β > 0 отнесена к bare-metal стенду (3.6).

## 3.5. Уточнение модели скоринга и его проверка

По результатам калибровки скор-функция расширена с сохранением исходной
структуры: вклад оси = `p_a · (α_a + β_a · s_задачи)`, где α — базовая
цена оси, β — надбавка чувствительной задачи; нормировка на
Σ(α_a + β_a). Пары (α, β) — пер-стендовые калибровочные константы
(замедление на единицу давления), продолжающие семейство калибровок
датчиков (`NET_REFERENCE_MBPS`, `LLC_REFERENCE_MISSES_PER_SEC`): датчики
нормируют *измерение* давления, цены осей — его *стоимость*. Модель
задачи Z = {G, R, S} не меняется: S остаётся переносимым свойством
задачи, стенд получает вектор цен.

Проверка на матрице токсичности смешанной серии (без новых прогонов):
исходный скоринг выбирает эмпирически лучший узел для 1 профиля из 3,
калиброванные веса при старой формуле — также 1 из 3, расширенный скоринг
α + β·s — 3 из 3.

Проверочная серия (повтор смешанного сценария с калиброванными ценами,
эталоны той же сессией, идентичные моменты прибытия; плечи default и
trimaran не зависят от весов и служат контролем дрейфа стенда) — критерии
успеха объявлены до получения данных:

1. кэш-профиль перестаёт проигрывать default (δ с +0.95 → ≤ 0);
2. выигрыши диск- и сеть-профилей сохраняются;
3. средний makespan SensitivityScore ≤ default.

[ОЖИДАЕТСЯ: результаты проверочной серии — таблица по трём критериям,
сравнение с исходной смешанной серией, вывод о статусе H1 после уточнения
модели.]

## 3.6. Ограничения и угрозы валидности

- **Облачный стенд**: суточный дрейф vCPU (контролируется правилом
  сессии), скрытая аппаратная неоднородность (контролируется пер-узловыми
  эталонами и пробами), PMU честен не у всех гипервизоров (контролируется
  perfcheck на Этапе 0).
- **CPU-квоты 500m** маскируют пер-задачное различение (β = 0): задачи
  не упираются в свои оси. Проверка β > 0 — на bare-metal с
  полноядерными квотами и/или утяжелёнными IO/Net-профилями.
- **Профиль high-s-net** не генерирует трафик — сетевая ось проверена
  по обнаружению и размещению, но не по деградации собственного обмена
  (запланирован стриминговый режим вывода).
- **Ось NUMA** на одноузловых машинах STAGE отключена честно; проверка —
  на многосокетном bare-metal.
- **Цена соседства** (γ) в скоринг пока не входит — оценена, оформлена
  как направление (прогноз давления с учётом вектора генерации G из
