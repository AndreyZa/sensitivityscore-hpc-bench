# Программа экспериментов на тестовом стенде (Geant4)

*PhD · Глава 1–2*

Сравнение архитектур планирования HPC-нагрузки (Geant4): K8s bare-metal, K8s+KubeVirt, Slurm, Slurm-on-K8s (Slinky)

---

## 1. Цель и контекст

Технические партнёры (группа учёных HPC-направления, hardware-профиль) подтвердили готовность развернуть тестовый стенд для проверки гипотез диссертации на реальной нагрузке. В качестве эталонного приложения выбран Geant4 — toolkit Монте-Карло симуляции взаимодействия частиц с веществом, относящийся к классу Particle Methods по таксономии Berkeley «13 Dwarfs» (Asanovic et al., 2006), как и LAMMPS.

Цель серии экспериментов — количественно подтвердить (или опровергнуть) гипотезу о том, что job-level, NUMA- и cache-aware планирование (плагин SensitivityScore) даёт измеримое преимущество по makespan и стабильности по сравнению с:

- (а) generic cloud-native планировщиками Kubernetes,
- (б) классическим HPC-планировщиком Slurm,
- (в) гибридными схемами совместного использования ресурсов K8s и Slurm (проект Slinky, SchedMD).

## 2. Тестовая нагрузка: Geant4

Geant4 удобен как эталонная нагрузка по нескольким причинам, значимым для вектора чувствительности S = {LLC, NUMA, Network, DiskIO}, формализованного в Главе 1:

- **LLC**: физические таблицы сечений взаимодействия (cross-section tables) и геометрия детектора создают существенный объём данных, к которому идёт нерегулярный доступ — ожидается высокая чувствительность к вытеснению из LLC при совместном размещении (co-location) с другими подами.
- **NUMA**: при многопоточном запуске (например, с MT-режимом Geant4 или MPI-обёрткой) рабочие потоки обращаются к общим read-only физическим таблицам — чувствительность к удалённым NUMA-обращениям растёт с числом потоков на сокет.
- **Network**: для распределённых прогонов (множество независимых «primary events», агрегация статистики) сетевая нагрузка умеренная и всплесковая — удобный контраст с задачами, чувствительными к сети (например, MPI-bound CFD).
- **Disk I/O**: запись выходных данных (hit collections, ntuple/ROOT-файлы) даёт управляемую I/O-нагрузку, которую можно дозировать через частоту и объём вывода.

Таким образом, Geant4 позволяет варьировать профиль чувствительности (через число потоков, physics lists, частоту записи) и проверять, насколько корректно SensitivityScore классифицирует и размещает job с разными профилями S.

## 3. Конфигурации тестового стенда

Технические партнёры предложили три базовых варианта развёртывания; ниже они расширены до четырёх (добавлен Slinky/slurm-bridge как промежуточный гибридный вариант, важный для главы о позиционировании работы относительно production-аналогов — Intel TAS, KubeAttention).

| Конфигурация | Слой планирования | Изоляция | Назначение в эксперименте |
|---|---|---|---|
| **A. K8s bare-metal** | kube-scheduler (default) vs SensitivityScore (custom plugin) | Linux namespaces / cgroups, контейнер | Базовая площадка для прямого A/B-сравнения предложенного плагина с дефолтным планировщиком |
| **B. K8s + KubeVirt** | kube-scheduler / SensitivityScore над KubeVirt VMI | Полная виртуализация (QEMU/KVM) поверх пода | Проверка применимости метода в VM-ориентированных HPC-кластерах; оценка накладных расходов виртуализации на S-метрики |
| **C. Slurm (классический)** | Slurm controller, без K8s | Процесс на выделенном узле (whole-node allocation) | Эталон HPC-производительности «как есть» — верхняя граница ожидаемой эффективности планирования |
| **D. Slinky (slurm-bridge)** | Slurm controller + K8s-scheduler-мост (co-location) | Под K8s, ресурсы транслируются в Slurm job | Проверка гибридной модели «Slurm как K8s-планировщик»; ближайший production-аналог совместного использования ресурсов |

### 3.1. Почему важно учитывать Slinky

По данным официальной документации (slinky.schedmd.com), проект Slinky выпускается SchedMD (разработчики Slurm) совместно с NVIDIA и состоит из двух компонентов:

- **slurm-operator** — разворачивает Slurm-кластер внутри Kubernetes (Slurm «как приложение на K8s»), допуская совместное использование ресурсов между Slurm и K8s.
- **slurm-bridge** — кастомный Kubernetes-планировщик, который делегирует выбор узла для части подов контроллеру Slurm; под из K8s транслируется в job-плейсхолдер, после allocation Slurm привязывает под к узлу, и далее им управляет штатный kubelet.

Это прямой аналог архитектурной идеи диссертации (job-level коммуникация между уровнем оркестрации и уровнем планирования), но без формальной модели задачи Z = {G, R, S} и без чувствительности к LLC/NUMA/IO — решение в slurm-bridge принимает фактически классический Slurm-планировщик по его политикам (приоритет, preemption, QoS, fairshare), не имея на входе профиля интерференции пода. Это усиливает формулировку научной новизны: slurm-bridge решает проблему совместного размещения организационно (через очередь Slurm), а не через модель интерференции.

**Технические ограничения slurm-bridge (важно для планирования эксперимента):**

- Минимальные версии: Kubernetes ≥ v1.35, Slurm ≥ 25.11 — нужно заложить в требования к стенду.
- Аллокации эксклюзивные, на целый узел (whole-node allocation) — не позволяет напрямую тестировать co-location нескольких job с разной чувствительностью на одном узле; для этого сценария подходят только конфигурации A и B.
- Поддерживаются только DRA-драйверы: `dra-driver-cpu` (CPU) и `dra-example-driver` (GPU) — нужно проверить совместимость с текущим стендом партнёров.

## 4. Метрики и их связь с вектором S

| Измерение S | Метрика (как реализовано) | Источник | В score? | Применимо к конфигурациям |
|---|---|---|---|---|
| LLC | узел: промахи/сек к калибровке `LLC_REFERENCE_MISSES_PER_SEC`¹; задача (fingerprint): miss ratio | `perf_event_open()` PMU, cgroup-scoped | да | A, C; B — ограниченно (см. §5) |
| NUMA | Доля remote-обращений (`node-load-misses / node-loads`) | `perf_event_open()`, generic NODE-события PMU | да | A, C; B — ограниченно |
| Network | `net_pressure = net_bw / NET_REFERENCE_MBPS` (+ сырой `net_bw`) | `/proc/<pid>/net/dev` (поды делят netns, hostPID-агент) | да² (после калибровки Этапа 0) | A, B, C, D |
| Disk I/O | `io_pressure` (PSI-доля ожидания IO, [0,1]) + сырые `io_iops` | cgroup v2 `io.pressure` (score) / `io.stat` (анализ) | да (PSI) | A, B, C, D |
| Интегральные | Makespan, slowdown, placement regret, throughput | terminated-времена контейнера / `sacct Elapsed` / снапшот давления | — | A, B, C, D |

Все четыре измерения S снимаются агентом и участвуют в score (STAGE
подтверждён; сверх того введена калибровка **цен осей** — весов score —
по вкладу оси в измеренное замедление, см. Сводку результатов).
¹ Для узла именно промахи/сек, а не отношение: под потоковой нагрузкой
знаменатель (обращения) растёт быстрее числителя, и загруженный узел
выглядит чище простаивающего — обнаружено smoke-проверкой на STAGE.
² Без выставленного референса агент пишет `net_pressure = 0` — ось
выключена честно, сырой `net_bw` остаётся для анализа. IO раздвоено:
`io_pressure` (нормировано ядром, в score) vs `io_iops` (сырое, только анализ).

**Калибровка Net (код реализован; на стенде остаётся измерить референс):** нормировать не на
номинальную скорость NIC (её и не прочитать честно — агент видит только
виртуальный `eth0` пода, у которого `/sys/class/net/*/speed` либо `-1`, либо
бессмысленное число; настоящий физический NIC хоста агенту не виден без
`hostNetwork: true`), а на **эмпирически измеренную через CNI** пропускную
способность pod-to-pod (`iperf3 --bidir` между двумя подами на **разных**
worker-нодах — тогда трафик реально уходит через физический uplink NIC, а не
мостится локально через veth, как на одной ноде; overlay CNI типа VXLAN
обычно даёт на 10-20%+ меньше номинала NIC). Результат — одно число
Мбит/с в переменную окружения агента (`NET_REFERENCE_MBPS`), партнёры
сообщают исходную топологию сети (см. §8), калибровка автоматизирована как
`make netcheck-run/-logs/-clean` (симметрично `perfcheck`), проводится на
Этапе 0 (см. §9), и уже прогнана вживую на 3-нодовом стенде. Дальше
`net_pressure = clamp(net_bw / (NET_REFERENCE_MBPS·1e6/8), 0, 1)` пишется в
Redis рядом с `net_bw`, ровно как `io_pressure` рядом с `io_iops`, и агент
подаёт его в score — этот код уже есть, включается выставлением env
`NET_REFERENCE_MBPS` на DaemonSet. Ограничения плана: один референс на тип узла (разнородное
железо потребует калибровки per-nodegroup — не в первой итерации); для
конфигурации B (KubeVirt) нужна отдельная калибровка на Этапе 3 — у
virtio-net/SR-IOV другой потолок пропускной способности, чем у pod-network A.

## 5. Методологическое ограничение: PMU-счётчики внутри VM (KubeVirt)

Это ключевой риск, который стоит обсудить с партнёрами до запуска серии B. `perf_event_open()` в текущем MVP читает счётчики hardware PMU напрямую с host. Внутри KubeVirt-машины доступ к PMU зависит от конфигурации виртуализации:

- По умолчанию KVM/QEMU не пробрасывает PMU гостю — perf-события внутри VM либо недоступны, либо показывают эмулированные/неточные значения.
- Частичное решение — vPMU passthrough (нужна поддержка в libvirt/QEMU и явное разрешение в KubeVirt VMI spec), но это снижает живую миграцию и плотность размещения VM на узле.
- Альтернатива для конфигурации B: собирать LLC/NUMA-метрики на host-уровне (cgroup, на которую отображается VM-процесс QEMU), что даёт агрегированный, но менее точный профиль на job.

**Рекомендация:** на первой итерации фиксировать в конфигурации B сбор LLC/NUMA не из гостя, а с host-стороны через перфоманс-счётчики процесса `qemu-kvm`, и явно пометить эту метрику в отчёте как приближение (`approximation: host-side`), отдельно обсудив с партнёрами доступность vPMU на их оборудовании.

## 6. Гипотезы

> Текущий статус проверки (по итогам девяти серий на стенде STAGE, июль
> 2026): H1 — по метрикам решения подтверждена; по времени подтверждена
> после уточнения модели (базовая цена оси), проверочная серия закрыла все
> заранее объявленные критерии; различение задач по чувствительности
> (cˢ > 0) доказано контрастом задач-близнецов с равными ресурсами на осях
> диска (×1.70) и сети (×6.11, плюс зазор по размещению). H1-trimaran —
> подтверждена. H2–H4 — ждут конфигураций B/C/D. Детали и цифры —
> «Сводка результатов STAGE (июль 2026)», секция «Статус гипотез».

- **H1** — SensitivityScore (конфигурация A) обеспечивает меньший makespan и меньшую дисперсию времени выполнения по сравнению с default kube-scheduler при совместном размещении job с высоким и низким профилем LLC/NUMA-чувствительности на одном узле.
- **H1-trimaran** (контроль) — преимущество SensitivityScore даёт именно вектор чувствительности S, а не «любой учёт загрузки узлов»: SensitivityScore обыгрывает и load-aware планировщик Trimaran (`LoadVariationRiskBalancing` — балансирует по утилизации CPU/памяти через metrics-server, но слеп к LLC/NUMA/IO-контенции) на pressure-сценариях, где агрессор с малым cpu-request невидим и для requests, и для утилизации. Отделяет interference-awareness от load-awareness.
- **H2** — Накладные расходы виртуализации (конфигурация B) снижают эффективность SensitivityScore по сравнению с A, но относительное преимущество SensitivityScore над default-планировщиком внутри B сохраняется (т.е. эффект от модели интерференции аддитивен к эффекту виртуализации).
- **H3** — Slurm (конфигурация C) как верхняя граница HPC-эффективности даёт меньший makespan на однородной нагрузке, но не учитывает интерференцию при искусственно созданном overcommit — то есть проигрывает SensitivityScore именно в сценариях co-location, а не в чистом batch-режиме.
- **H4** — Slinky/slurm-bridge (конфигурация D) воспроизводит политику Slurm (приоритет, fairshare), но без профиля интерференции по S; ожидается, что по интегральным метрикам D близок к C на однородной нагрузке, но проигрывает A в сценариях смешанной чувствительности — это эмпирически подтверждает научную новизну (job-level + interference-aware модель против организационного разделения ресурсов).

## 7. План эксперимента (факторы и контроль)

### 7.1. Независимые переменные

- Конфигурация стенда: A / B / C / D
- Профиль чувствительности нагрузки: low-S (однопоточный батч, малый physics list) / high-S (MT, полный physics list, частая запись)
- Коэффициент совместного размещения (overcommit ratio): 1.0 (изолированно) / 1.5 / 2.0 job на узел

### 7.2. Зависимые переменные

- Makespan job (медиана, p95) и **slowdown** = makespan / изолированный (соло-бейзлайн) — безразмерная нормировка, объединяет профили разной длительности
- **Placement regret** — давление выбранной планировщиком ноды минус минимально доступное (прямая метрика качества решения, почти без шума исхода)
- LLC miss ratio, доля NUMA remote-обращений, IO `io_pressure` (PSI) — где доступно
- Дисперсия времени выполнения между повторами (стабильность, CV)

### 7.3. Контроль и повторения

- Фиксированная версия Geant4 и физических списков на всех конфигурациях
- ≥ 10 повторов на точку плана для статистической значимости (с учётом шума HPC-узлов)
- Прогрев узлов (warm-up run) перед измерением, исключение влияния холодного кэша файловой системы
- Фиксация версий K8s, KubeVirt, Slurm, slurm-bridge на момент серии экспериментов (с учётом версионных требований slurm-bridge ≥ K8s 1.35 / Slurm 25.11)

## 8. Открытые вопросы техническим партнёрам

> **Прод-стенд подтверждён:** 3× Dell PowerEdge R760, каждый — 2× Intel Xeon
> Platinum 8462Y+ (Sapphire Rapids, 32c/64t на сокет → 64c/128t, **2 сокета /
> 2 NUMA-домена** на узел), 384 ГБ RAM, **bare-metal**. Это закрывает часть
> вопросов ниже: топология NUMA известна (NUMA-ось активна), железо однородно,
> сбор PMU для A/C честный (нет гипервизора) — vPMU остаётся вопросом только
> для B (KubeVirt).

- Какая версия Kubernetes будет на стенде — совместима ли она с минимальными требованиями slurm-bridge (≥ 1.35), если конфигурацию D включаем в первую итерацию?
- Поддерживает ли оборудование vPMU passthrough для KubeVirt (нужно для не-приближённого сбора LLC/NUMA метрик в конфигурации B)? (Хост bare-metal → для A/C сбор честный; вопрос только про гостевые VM конфигурации B.)
- ~~Какова топология NUMA на узлах стенда~~ — **уточнено:** 2 сокета / 2 NUMA-домена на узел (Xeon 8462Y+), NUMA-ось активна; влияет на чувствительность плана к overcommit ratio.
- Возможна ли установка кастомного scheduler plugin (SensitivityScore) с правами на чтение PMU (CAP_PERFMON) на узлах стенда, либо стенд изначально ограничен политиками безопасности?
  **Уточнение по итогам отладки:** нужно, чтобы `perf_event_open()` работал в **cgroup-scoped** режиме (`PERF_FLAG_PID_CGROUP`), который использует агент. Долгое время cgroup-scoped открытие падало с `EINVAL` (в т.ч. под `--privileged` и при `perf_event_paranoid=-1`), и это ошибочно списывалось на гипервизор (WSL2/Docker Desktop/облако). На деле это был **баг в нашем коде**: cgroup-события измеряются по-CPU и требуют `cpu >= 0`, а агент открывал счётчик с `cpu = -1` → `EINVAL` в любой среде. После фикса (открытие по одному счётчику на online-CPU) cgroup-scoped PMU **реально работает и считает** — подтверждено на WSL2 и, что важнее, на реальном стенде: perfcheck дал ~3.0–3.3M cache-misses на всех трёх новых bare-metal нодах (worker-192.168.0.8/9/10) И на облачной STAGE-VM worker-192.168.0.5. На VMware Workstation Pro, для контраста, open теперь проходит, но счётчик читается нулём: её vPMU, похоже, не виртуализирует cgroup-scoped-подсчёт кэш-событий (per-process `perf stat` при этом что-то считает). Вывод: «виртуализация ⇒ нули» НЕверно в общем случае — это зависит от гипервизора: Timeweb-облако (KVM) виртуализирует cgroup-PMU честно, VMware Workstation — нет. Именно поэтому `perfcheck` гоняется поузлово (open OK, read=0 → inconclusive; open OK, read>0 → честно). Диагностический инструмент `metrics-agent/cmd/perfcheck` по-прежнему стоит прогнать первым шагом на стенде (см. Этап 0) — он ловит уже не наш баг, а честность конкретного железа (гипервизор, подделывающий счётчики нулями, и т.п.).
- Планируется ли GPU-нагрузка позже — если да, стоит сразу учитывать ограничение slurm-bridge по поддерживаемым DRA-драйверам (`dra-example-driver` для GPU)?
- Включён ли в ядре стенда **PSI** (`CONFIG_PSI=y`, cgroup v2 `io.pressure`)? Это новый пункт: IO-измерение вектора S теперь строится на PSI-доле времени ожидания IO (честная величина в [0,1] из самого ядра), а не на сырых IOPS — без PSI это измерение останется нулевым (агент один раз предупредит в логе, но не упадёт). Проверяется тривиально: `cat /sys/fs/cgroup/io.pressure` на любом узле стенда.
- Установлен ли на стенде **metrics-server**? Нужен только для опционального контрольного плеча Trimaran (H1-trimaran) — оно читает утилизацию узлов через `metrics.k8s.io`. Если нет и плечо нужно — ставится в одну команду (`make trimaran-deps`); на основной эксперимент (default vs SensitivityScore) не влияет.
- Какова топология сети узлов стенда — скорость NIC, один физический интерфейс на узел или несколько (агрегация/bonding), одна ли модель узла везде или сеть неоднородна? Нужно для калибровки Net-измерения (см. §4, план калибровки): референсное значение для нормировки `net_bw` берётся не из номинала NIC (не читается честно из-под CNI), а из эмпирического `iperf3 --bidir` cross-node теста (между двумя разными worker-нодами) на Этапе 0 — топология влияет на то, нужен ли один референс на весь стенд или per-nodegroup.
