# Ввод прод-стенда — чек-лист Этапа 0

Что проверить и зафиксировать при развёртывании нового прод-стенда, до
запуска полноценной серии. Порядок — от самого важного (снимает главные
риски за 10 минут) к справочному. Подробное обоснование метрик — в
`Программа экспериментов (Geant4).md` §4/§8 и `Технический план
экспериментов.md` §3.

Контекст: мы сравниваем несколько схем планирования HPC-нагрузки (K8s
bare-metal, K8s+KubeVirt, Slurm, Slinky) на Geant4. Ключевая метрика
опирается на аппаратные счётчики (LLC/NUMA через PMU) и давление по IO/сети —
поэтому доступность этих счётчиков на железе стенда определяет, будет ли сбор
честным или придётся помечать данные как приближение.

## 1. Первый шаг — два готовых диагностических прогона (10 минут)

Запускаются одним подом каждый, сразу показывают, честный ли будет сбор.
Прогнать на КАЖДОМ типе узла до разворачивания основного агента.

### (а) PMU — cgroup-scoped `perf_event_open()`

Агент использует **cgroup-scoped** режим (`PERF_FLAG_PID_CGROUP`). Если
cgroup-scoped открытие падает с `EPERM`/`permission denied` — это права: на
hardened-ядрах (`perf_event_paranoid ≥ 3`) одного `CAP_PERFMON` может не
хватать, нужен либо sysctl пониже, либо `CAP_SYS_ADMIN`. Если падает с
`EINVAL` — это НЕ среда: cgroup-события требуют `cpu >= 0` (агент это
учитывает, открывая счётчик на каждый online-CPU); раньше здесь был баг
`cpu=-1`, дававший `EINVAL` везде, — теперь исправлен. `perfcheck` гоняем,
чтобы подтвердить, что счётчики на железе стенда честно считают (не нули от
гипервизора), а не для проверки нашего кода.

```bash
make perfcheck-image
make perfcheck-push                      # узлы тянут образ из registry
make perfcheck-run NODE=<имя-узла>       # на неоднородном стенде — по каждому узлу; без NODE встаёт куда придётся
make perfcheck-logs                      # ждём STATUS Completed; в логе SUCCESS с ненулевым числом
make perfcheck-clean
```

Три исхода: `SUCCESS` + ненулевое число = PMU честный (LLC/NUMA точны);
`WARNING ... exactly zero` = счётчик открылся, но не считает (гипервизор
подделывает — неубедительно); `FAILED to open perf counter` =
`perf_event_open` заблокирован.

### (б) Сеть — реально достижимая cross-node полоса

Для нормировки сетевого измерения нужен эмпирический референс — не номинал
NIC (из-под CNI его честно не прочитать), а реально достижимая полоса между
**разными** узлами (same-node трафик мостится локально через veth, минуя
физический NIC).

```bash
make netcheck-run                        # iperf3 между двумя разными worker-нодами
make netcheck-logs                       # выводит строку NET_REFERENCE_MBPS=<число>
make netcheck-clean
make netcheck-apply NET_REFERENCE_MBPS=<N>   # выставить референс на DaemonSet агента
```

Прогнать на нескольких парах узлов: если полоса неоднородна (на одном из
тестовых стендов «одинаковые» узлы дали 900–1335 Мбит/с, ~1.5×), это влияет
на выбор единого референса vs per-nodegroup. `make netcheck-apply`
перезапускает агент — делать МЕЖДУ сериями, не посреди прогона; `make
netcheck-disable` выключает Net-ось (сырой `net_bw` всё равно пишется, для
анализа).

## 2. Версии ПО (зафиксировать для воспроизводимости)

- **Kubernetes** — версия. Для конфигурации D (Slinky/slurm-bridge) нужны
  K8s ≥ 1.35 и Slurm ≥ 25.11; если ниже — D откладываем, на A/B/C не влияет.
- **Slurm** и (если применимо) **KubeVirt** — версии на момент серии.

## 3. Железо

- **Топология NUMA**: сколько сокетов и NUMA-доменов на узел. Влияет на
  чувствительность плана к overcommit — на односокетных узлах NUMA-ось теста
  вырождается (удалённых обращений нет).
- **Топология сети**: скорость NIC, один физический интерфейс или несколько
  (bonding), одинаковое ли железо на всех узлах (см. netcheck п.1б).

## 4. Ядро и политики безопасности

- **PMU-доступ**: разрешена ли установка кастомного scheduler-плагина и агента
  с `CAP_PERFMON` на узлах (perfcheck из п.1 проверяет фактическую
  работоспособность; здесь — формальное «можно»). На hardened-ядрах уточнить
  `perf_event_paranoid` и не требуется ли `CAP_SYS_ADMIN` для cgroup-scoped.
- **PSI** (`CONFIG_PSI=y`, cgroup v2) — IO-измерение строится на PSI-доле
  времени ожидания IO. Проверка тривиальна на любом узле:
  ```bash
  cat /sys/fs/cgroup/io.pressure
  ```
  Нет файла / PSI выключен → IO-ось останется нулевой (агент предупредит, но
  не упадёт).
- **cgroup v2** (unified hierarchy) — агент рассчитан на v2, не v1:
  ```bash
  stat -fc %T /sys/fs/cgroup       # cgroup2fs = v2 (нужно), tmpfs = v1/hybrid
  ```

## 5. На будущее (не блокирует старт)

- **Конфигурация B (KubeVirt)**: поддерживает ли оборудование vPMU
  passthrough для гостевых VM. Без него LLC/NUMA внутри VM недоступны/неточны
  — тогда собираем host-side и помечаем как приближение. Важная эмпирика:
  «виртуализация ⇒ нет PMU» неверна в общем случае. Cgroup-scoped perf,
  который нужен агенту, честно считает на STAGE-VM Timeweb (KVM) — perfcheck
  дал ~3.1M cache-misses на worker-192.168.0.5, столько же, сколько на
  bare-metal .8/.9/.10. Для контраста VMware Workstation Pro пробрасывает
  per-process `perf stat` в гость, но cgroup-scoped-подсчёт кэша ей читается
  нулём. Т.е. поведение зависит от конкретного гипервизора — потому perfcheck
  из п.1 и гоняется поузлово, а не выводится из «это VM».
- **GPU**: если планируется — у slurm-bridge есть ограничения по
  поддерживаемым DRA-драйверам, учесть заранее.
