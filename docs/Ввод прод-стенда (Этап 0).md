# Ввод прод-стенда — чек-лист Этапа 0

Что проверить при развёртывании прод-стенда до запуска серии. Порядок — от
критичного к справочному. Обоснование метрик — `Программа экспериментов
(Geant4).md` §4/§8 и `Технический план экспериментов.md` §3.

Ключевые метрики опираются на аппаратные счётчики (LLC/NUMA через PMU) и
давление IO/сети; доступность счётчиков на узле определяет, будет сбор честным
или приближением.

## 1. Диагностические прогоны (10 минут)

По одному поду каждый, на КАЖДОМ типе узла до разворачивания агента.

### (а) PMU — cgroup-scoped `perf_event_open()`

Агент использует **cgroup-scoped** режим (`PERF_FLAG_PID_CGROUP`).
`EPERM`/`permission denied` — права: на hardened-ядрах
(`perf_event_paranoid ≥ 3`) `CAP_PERFMON` может не хватать, нужен sysctl ниже
(§5) или `CAP_SYS_ADMIN`. `EINVAL` — не среда: cgroup-события требуют
`cpu >= 0`, агент открывает счётчик на каждый online-CPU. perfcheck
подтверждает, что счётчики честно считают, а не отдают нули от гипервизора.

```bash
make perfcheck-image
make perfcheck-push                      # узлы тянут образ из registry
make perfcheck-run NODE=<имя-узла>       # на неоднородном стенде — по каждому узлу; без NODE встаёт куда придётся
make perfcheck-logs                      # ждём STATUS Completed; в логе SUCCESS с ненулевым числом
make perfcheck-clean
```

Исходы: `SUCCESS` + ненулевое число = PMU честный (LLC/NUMA точны);
`WARNING ... exactly zero` = счётчик открылся, но не считает (гипервизор
подделывает); `FAILED to open perf counter` = `perf_event_open` заблокирован.

### (б) Сеть — cross-node полоса

Референс для нормировки — не номинал NIC (из-под CNI его не прочитать), а
достижимая полоса между **разными** узлами (same-node трафик мостится через
veth, минуя физический NIC).

```bash
make netcheck-run                        # iperf3 между двумя разными worker-нодами
make netcheck-logs                       # выводит строку NET_REFERENCE_MBPS=<число>
make netcheck-clean
make netcheck-apply NET_REFERENCE_MBPS=<N>   # выставить референс на DaemonSet агента
```

Прогнать на нескольких парах узлов: неоднородная полоса (наблюдалось
900–1335 Мбит/с между «одинаковыми» узлами) влияет на выбор единого референса
vs per-nodegroup. `make netcheck-apply` перезапускает агент — между сериями,
не в прогоне; `make netcheck-disable` выключает Net-ось (сырой `net_bw`
пишется всегда, для анализа).

## 2. Разметка узлов ролями и системные компоненты

Системные компоненты (Redis метрик, sensitivityscore-scheduler с профилем
trimaran, metrics-server) — на выделенных узлах рядом с control-plane, отдельно
от измерительных: (а) не шумят в LLC/IO/Net измерительных узлов, (б) переживают
пересборку экспериментальной части кластера.

Разметка — ролями `node-role.kubernetes.io/*` (колонка ROLES `kubectl get
nodes`); харнесс и статус-сервер выбирают измерительные узлы по ролям, ad-hoc
лейблы не используются:

```bash
# один системный узел/ВМ (можно несколько аргументов):
./scripts/bootstrap-cluster.sh <имя-системного-узла>
```

Скрипт идемпотентен: роль `ss-system` + taint
`node-role.kubernetes.io/ss-system:NoSchedule` системным узлам; роль `bench`
остальным worker-узлам; неймспейс; Redis из `k8s/scheduler-config/redis.yaml`.
Taint не пускает задачи эксперимента на системный узел ни под каким
планировщиком.

Топология: три R760 — измерительные (`bench`), все три полностью под
эксперимент; control-plane и `ss-system` — на служебных ВМ (см. §4, отдельный
гипервизор-хост, НЕ на R760). `bootstrap-cluster.sh` получает имя
ss-system-ВМ. Дальше — `make scheduler-apply-config scheduler-deploy`,
`make trimaran-deps` (+ перевод metrics-server на системный узел:
`kubectl -n kube-system patch deployment metrics-server
-p '{"spec":{"template":{"spec":{"nodeSelector":{"node-role.kubernetes.io/ss-system":""},
"tolerations":[{"key":"node-role.kubernetes.io/ss-system","operator":"Exists","effect":"NoSchedule"}]}}}}'`),
агент DaemonSet'ом и две калибровки датчиков через `kubectl set env`:
`NET_REFERENCE_MBPS` (из netcheck, §1б) и `LLC_REFERENCE_MISSES_PER_SEC`
(misses/s из поля `llc_misses_per_sec` узла под эталонным штормом
2×stress-ng --stream).

Третья калибровка — **цены осей** (веса score) — выполняется после первых
серий на стенде: `analysis/calibrate_axis_costs.py` оценивает вклад каждой
оси в измеренное замедление (базовая и чувствительностная части), результат
кладётся в ConfigMap `sensitivity-config` в формате
`weights.json = {"base": {…}, "sensitivity": {…}}` + rollout restart
планировщика (в логе появится «sensitivity weights loaded»). До калибровки
допустимы равные веса — но выводы о времени выполнения до неё делать
нельзя (урок STAGE: цены осей различаются на порядок).

Урок STAGE для сетевой оси: шторм для измерения деградации (не только
детекции) обязан быть **egress** — iperf3-клиенты на штормовом узле, сервер
на другом (`mode: net-egress` в конфиге серии); локальная пара на одном узле
мостится через veth, аплинк не занимает и обмен соседа не замедляет
(проба: `scripts/net_probe.sh`). Перед сетевой серией на новом стенде
повторить пробу — на другом железе/CNI соотношение может отличаться.

## 3. Версии ПО (зафиксировать для воспроизводимости)

## 4. Железо

Прод-стенд: **3× Dell PowerEdge R760**, каждый — 2× Intel Xeon Platinum
8462Y+ (Sapphire Rapids, 32 ядра / 64 потока на сокет → 64 ядра / 128 потоков
и **2 NUMA-домена** на узел), 384 ГБ RAM, bare-metal (без гипервизора).

Следствия для теста (отличия от STAGE):

- **NUMA-ось активна** — 2 сокета/узел против одно-NUMA STAGE: MT-задачи с
  потоками за границей сокета дают ненулевой `numa_remote_ratio`.
- **LLC**: L3 у Xeon socket-wide (не CCX-группы, как у EPYC) — узловой датчик
  видит контраст L3 сокета целиком, без разбавления.
- **PMU честный** — bare-metal, гипервизор счётчики не подделывает; `perfcheck`
  (§1) это подтвердит. vPMU-вопрос остаётся только для конфигурации B (KubeVirt).
- **Полные ядра** (не 500m-квоты STAGE) — задачи перестают быть чисто
  compute-bound, и различение по декларации `S` (β > 0) становится проверяемым.
  `G4_THREADS` профиля high-s = число ядер NUMA-домена (32), env-оверрайдом.

- **Топология сети**: скорость NIC, один интерфейс или bonding, одинаковое ли
  железо на узлах (см. netcheck §1б).

### Служебные ВМ (control-plane + ss-system)

Четыре ВМ по **4 vCPU / 8 ГБ / 100 ГБ SSD** на отдельном гипервизор-хосте
(**НЕ на R760** — иначе фоновая нагрузка etcd/ClickHouse исказит метрики
bench-узлов):

- **3× control-plane** (HA-кворум etcd): apiserver, etcd, controller-manager,
  default-scheduler. Узкое место — латентность fsync диска etcd (p99 < 10 мс),
  поэтому диск SSD/NVMe.
- **1× ss-system**: Redis (метрик-бэкенд, hot-path планировщика),
  SensitivityScore-scheduler, metrics-server, in-cluster ClickHouse (агрегатор
  результатов — драйвер диска; Redis + scheduler + metrics-server вместе <1 ГБ).
  Размечается ролью `ss-system` + taint (§2).

Альтернатива: managed control-plane (как Timeweb k0s на STAGE) — тогда 3 CP-ВМ
не нужны, остаётся одна ss-system-ВМ.

## 5. ОС, ядро и политики безопасности

### ОС

**Ubuntu Server 24.04 LTS** (или 22.04 LTS). Причины — по метрикам стенда:

- **PSI включён по умолчанию** — от него зависит IO-ось (`io.pressure`). У
  Debian 12 и части RHEL-сборок PSI выключен (`CONFIG_PSI_DEFAULT_DISABLED`),
  нужен boot-параметр `psi=1`.
- **cgroup v2 (unified)** по умолчанию — агент рассчитан на неё.
- **Ядро 6.8** → `CAP_PERFMON` (нужно ≥5.8) и честный cgroup-scoped PMU.
- Совместимость с k0s/kubeadm/Calico/Cilium; LTS = 5 лет поддержки.

Альтернативы: **Rocky/AlmaLinux 9** (проверить PSI, возможно `psi=1`);
**Debian 12** (добавить `psi=1`). **Immutable (Talos/Flatcar)** — нет
shell/пакетного менеджера, а стенду нужно узловое перф-ковыряние (perfcheck,
тюнинг paranoid).

### `perf_event_paranoid` (навсегда)

С `CAP_PERFMON` дефолт `=2` часто достаточно — сначала perfcheck. При `FAILED
to open` понизить sysctl и закрепить (в `/etc/sysctl.d/`, иначе `echo` в
`/proc/...` живёт до ребута):

```bash
echo 'kernel.perf_event_paranoid = 1' | sudo tee /etc/sysctl.d/99-perf.conf
sudo sysctl --system                         # применить без ребута
cat /proc/sys/kernel/perf_event_paranoid     # проверка -> 1
```

Настройка узловая (не подовая) — на каждом воркере с агентом; для прода
зашивать в провижининг (cloud-init/Ansible/образ). `paranoid=3` — downstream-
хардненинг (в mainline максимум 2): даже `CAP_PERFMON` может не хватить, тогда
sysctl ниже или `CAP_SYS_ADMIN`.

### PSI и cgroup v2

- **PSI** (`CONFIG_PSI=y`) — IO-измерение строится на PSI-доле времени
  ожидания IO:
  ```bash
  cat /sys/fs/cgroup/io.pressure          # или /proc/pressure/io
  ```
  Нет файла / PSI выключен → IO-ось нулевая (агент предупредит, не упадёт). На
  Debian/RHEL без `psi=1` файл пустой/отсутствует.
- **cgroup v2** (unified hierarchy) — агент рассчитан на v2, не v1:
  ```bash
  stat -fc %T /sys/fs/cgroup       # cgroup2fs = v2 (нужно), tmpfs = v1/hybrid
  ```

## 6. На будущее (не блокирует старт)

- **Конфигурация B (KubeVirt)**: есть ли vPMU passthrough для гостевых VM. Без
  него LLC/NUMA внутри VM недоступны/неточны — собираем host-side как
  приближение. «Виртуализация ⇒ нет PMU» неверно в общем случае: зависит от
  гипервизора (Timeweb KVM cgroup-PMU считает честно, VMware Workstation —
  нулём), поэтому perfcheck гоняется поузлово.
- **GPU**: у slurm-bridge ограничения по поддерживаемым DRA-драйверам.
