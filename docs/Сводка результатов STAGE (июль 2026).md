# Сводка результатов STAGE (июль 2026)

Термины (шторм, жертва, токсичность, ошибка размещения…) — в
[Глоссарии](<Глоссарий.md>).

**Итог.** Восемь серий: диск, сеть, кэш, смешанная (три шторма — группы
генераторов фоновой нагрузки, по одной оси на узел — одновременно),
смешанная с калиброванным скорингом, плацебо (без нагрузки),
io-sensitivity (различение по чувствительности к диску), differentiation
(изоляция cˢ_io двойником с равными ресурсами);
3 планировщика × 10 повторений.
По метрикам решения SensitivityScore лучший во всех сериях с нагрузкой.
По времени — выигрыш на дорогих осях; два проигрыша на дешёвых вскрыли
ограничение модели (нет цены оси). Калибровка дала уточнение — вклад оси =
давление·(α + β·чувствительность) — и отдельная серия закрыла все три
заранее объявленных критерия проверки. Плацебо-серия различий между
планировщиками не показала — выигрыши не артефакт методики. Серия
io-sensitivity показала различение задач по чувствительности к диску
(cˢ_io ≈ 0.63 > 0): под штормом чувствительная задача замедляется ×1.86,
нечувствительная ×1.23 — прежнее «cˢ = 0» было артефактом нагрузки, не
стенда. Серия differentiation изолировала надбавку на равных ресурсах:
контраст двойников (io-чувствительный vs идентичный нечувствительный) на
одном штормовом узле дал ×1.70 (p = 3.8·10⁻⁸), и только SensitivityScore
удержал чувствительную задачу на базовом времени, тогда как load-aware
trimaran паковал её на шторм и вредил.

Стенд: Timeweb managed k8s (k0s), системные компоненты на выделенном узле
(роль `ss-system`). Узлы: 2 vCPU (KVM, ~3,1 ГГц), 1,9 ГиБ RAM, том 38 ГиБ.
Жертва: Geant4, 300 тыс. частиц, 2 потока, квота 500m CPU; шторм = 2 пода
генераторов на узел (цифры по осям — в [Методике](<Методика измерений.md>)).
Серии кэша и смешанные — на PMU-тройке w8/w9/w10
(эталоны однородны, 68–74 с ±2–3 %); диск и сеть — на раннем составе из
6 узлов, включая ~2× медленный w7 (поглощается эталонами на каждом узле).
Планировщики: default, SensitivityScore, trimaran
(`LoadVariationRiskBalancing` — утилизация CPU/памяти, без модели
интерференции). Статистика: Манн-Уитни по повторениям, поправка Холма,
Cliff's δ. Данные всех серий — в ClickHouse, каждая запись помечена
стендом и серией (провенанс); отчёт воспроизводится: `make ch-tunnel &&
make ch-analyze STAND=stage RUN_LABEL=<серия>`.

## Статус гипотез

- **H1** — по метрикам решения подтверждена во всех сериях; по времени
  подтверждена на дорогих осях, опровергнута в исходной форме скоринга на
  дешёвых; после уточнения модели проверочная серия систематических
  проигрышей не оставила. Различение задач по чувствительности (cˢ > 0)
  **показано на дисковой оси** двумя сериями: io-sensitivity (под диск-
  штормом чувствительная задача ×1.86 vs нечувствительная ×1.23) и, чище,
  differentiation — контраст двойников с **равными ресурсами и compute** на
  одном штормовом узле дал ×1.70 (cˢ_io ≈ 0.63, p = 3.8·10⁻⁸), устранив
  конфаунд упаковки; надбавка cˢ_io отделена от базовой c⁰_io (нечувствительный
  двойник на шторме лишь ×1.04). Там же показано, что на дисковой оси **только**
  SensitivityScore защищает чувствительную задачу (медиана 92 с = база), тогда
  как load-aware trimaran паковал её на шторм (157.5 с). Проверка cˢ_llc — на
  проде (квоты 500m ещё маскируют кэш-ось).
- **H1-trimaran** — подтверждена: load-aware проигрывает по решениям
  везде; в смешанных сериях систематическая ошибка (до 35/60 задач на
  дисковый узел, «свободный» по CPU).
- **H2–H4** — не проверялись (нужны конфигурации B/C/D).

Методические результаты по пути: инверсия отношения промахов кэша под
потоковой нагрузкой (датчик переведён на промахи/сек с калибровкой);
суточный дрейф облачных vCPU +13–23 % → правило «эталоны в ту же сессию»;
эталоны на каждом узле против скрытой неоднородности (узлы бывают ~2× разными);
плацебо-контроль — без нагрузки различий между планировщиками нет.

## Почему проигрывал исходный скоринг: цены осей

Матрица токсичности смешанной серии (время профиль × узел):

| профиль | w8 (кэш-шторм) | w9 (диск-шторм) | w10 (сеть-шторм) |
|---|---|---|---|
| кэш-чувствительный | **77.9 с** | 96.2 с | 80.6 с |
| диск-чувствительный | 82.4 с | **97.0 с** | 85.4 с |
| сеть-чувствительный | 78.5 с | 94.6 с | 80.9 с |

Дисковый шторм замедляет все профили (+15–18 с); кэш-шторм при квотах 500m практически не влияет.
Калибровка по 360 задачам (модель «замедление ≈ γ·соседи + Σ p·(α + β·s)»,
NNLS, посерийный интерсепт; R² по ячейкам 0.88):

| ось | α — базовая цена (платят все) | β — надбавка чувствительной |
|---|---|---|
| диск | **+0.24** | 0 |
| кэш | 0 | 0 |
| сеть | +0.02 | 0 |

γ (сосед) = +0.026; дрейф сессии +0.13…+0.23 — крупнейший член. Следствия:
(1) цена оси базовая — скоринг Σ w·s·p её не выражает (задача с s = 0 не
платит при любых весах), перевзвешивание не лечит; (2) на матрице
токсичности исходный скоринг выбирает лучший узел для 1 профиля из 3,
расширенный p·(α + β·s) — для 3 из 3; (3) β = 0 в этой калибровке — не
свойство стенда, а артефакт нагрузки: дисковый вывод жертвы шёл фоновым
процессом, умиравшим с контейнером, и makespan от диска не зависел. С
выводом на критическом пути (серия io-sensitivity ниже) cˢ_io > 0
измеряется прямо.

## Различение по чувствительности cˢ_io > 0 — серия io-sensitivity (15.07)

Прежняя калибровка дала cˢ = 0 не из-за стенда, а из-за нагрузки:
дисковый вывод жертвы шёл фоновым процессом, умиравшим с контейнером, —
makespan от диска не зависел. Воркер доработан: вывод сбрасывается на
диск **на критическом пути** (`OUTPUT_MODE=blocking`: compute, затем
fsync-сброс; job не готов, пока результаты не персистентны). Замер: диск-
шторм режет пропускную запись соседа ×23 (голый dd 2.5 ГБ: 6 с → 138 с).

Серия: диск-шторм на одном узле, жертвы high-s-io (вывод на диск) и low-s
(без вывода) поровну, 3 планировщика × 10 повторений, 180/180 без ошибок.

| На штормовом узле | slowdown | n |
|---|---|---|
| high-s-io (чувствительная) | **1.86** | 11 |
| low-s (нечувствительная) | 1.23 | 25 |

Отсюда c⁰_io ≈ 0.23 (базовая, платит и нечувствительная), **cˢ_io ≈ 0.63**
(надбавка чувствительной); high-s-io медленнее low-s под штормом
p < 10⁻⁴ (Манн-Уитни). На чистых узлах различия нет (1.01 vs 1.03) —
контроль. Матрица токсичности: high-s-io 90 с (чисто) → 160 с (шторм),
low-s 38 → 48 с.

Размещение: trimaran (load-aware) поставил 11/30 high-s-io на штормовой
узел — по CPU он «свободен» — и погубил их (среднее время 115 с против
89). SS и default обе жертвы на w9 не ставили (0/30 high-s-io каждый).
**Осторожно с интерпретацией:** у default это не «ум», а артефакт упаковки —
профили имели разные ресурсные заявки (low-s 1 ГиБ, high-s-io 384 МиБ
памяти), и профиль-слепой планировщик развёл их по узлам случайно
относительно интерференции (узел w9 при этом ни разу не получил
disk-pressure-taint — проверено). Поэтому серия чисто доказывает **физику**
(cˢ_io > 0) и **вред load-aware** (trimaran), но НЕ изолирует преимущество
SS над слепым default: для этого нужны равные ресурсы профилей (чтобы
default клал их одинаково) и веса с cˢ_io > 0 (чтобы SS уводил чувствительную
сильнее нечувствительной) — серия differentiation (запущена, см. ниже).

Следствие: различение задач по чувствительности (cˢ > 0) — не «только для
прода». На дисковой оси физика показана на STAGE; для кэш-оси квоты 500m её
ещё маскируют — это прод.

## Изоляция cˢ_io двойником — серия differentiation (15.07)

Веса перекалиброваны по io-sensitivity (base io = 0.23, sens io = 0.63;
ConfigMap пропатчен, плагин перечитал). Добавлен профиль-двойник
**io-insensitive** — идентичен high-s-io по ресурсам и compute (500m CPU,
2 треда, 300k, 384 МиБ), но без дискового вывода и с io = low. Обе жертвы
поровну под тем же диск-штормом (worker w9), ресурсы равны — значит любое
различие между ними принадлежит **только** чувствительности к диску, а не
упаковке по ресурсам (тот конфаунд, что размывал io-sensitivity).
3 планировщика × 10 повторений, 180/180 без ошибок.

**cˢ_io доказан контрастом двойников (главное).** На **одном и том же**
штормовом узле, при равных compute и ресурсах, чувствительный двойник
медленнее нечувствительного в **1.70 раза** (медиана 160 с против 94 с,
n = 29/15, Манн-Уитни p = 3.8·10⁻⁸). Единственное различие — io-
чувствительность и реальный вывод на диск. Это изолирует надбавку cˢ_io
чище, чем io-sensitivity: там low-s отличался и compute, и ресурсами.

| На штормовом узле (равные ресурсы) | makespan (медиана) | slowdown vs чисто |
|---|---|---|
| high-s-io (io = high, вывод на диск) | **160 с** | **×1.74** |
| io-insensitive (io = low, тот же compute) | 94 с | ×1.04 |

Разложение (c⁰ + cˢ·s)·p видно напрямую: нечувствительный двойник на штормовом
узле замедляется лишь ×1.04 (94 против 90 с, p = 2·10⁻⁴) — это базовая цена
c⁰_io, которую платят все через iowait; чувствительный — ×1.74, разница и есть
надбавка cˢ_io. На чистых узлах оба на базовом уровне (91–92 с) — контроль.

**SS — единственный, кто защищает чувствительную задачу.** high-s-io по
планировщикам (медиана makespan / средний регрет):

| Планировщик | high-s-io makespan | регрет | high-s-io на шторме |
|---|---|---|---|
| **SensitivityScore** | **92 с** (= база) | **0.002** | **0 / 30** |
| default | 96.5 с (p = 3.5·10⁻³) | 0.273 | 9 / 30 |
| trimaran | 157.5 с (p = 1.9·10⁻⁶) | 0.601 | 20 / 30 |

SS держит high-s-io на базовом уровне; trimaran (load-aware) наоборот **пакует**
чувствительную задачу на штормовой узел (по CPU он «свободен») и губит её,
уступая даже слепому default.

**Честно о размещении.** Предрегистрированное ожидание было «у SS high-s-io
попадает на шторм *реже* двойника» (различение по размещению). Такого зазора
не возникло: SS увёл со шторма **обе** жертвы (0 / 30 каждая). Причина — при
двух свободных узлах базовой цены c⁰_io уже достаточно, чтобы увести со шторма
любую пишущую на диск задачу; надбавке cˢ_io не пришлось решать, «кем
пожертвовать». Различение проявилось не в размещении, а в **makespan** (контраст
двойников) — это более фундаментальная подпись cˢ_io. Зазор по размещению
проявился бы при дефиците чистой ёмкости (свободных узлов < жертв) — это дизайн
следующей серии. default остался профиль-слепым (30 % против 20 % — статистически
шум), что подтверждает: его нули в io-sensitivity были артефактом упаковки, а не
учётом интерференции.

Итог differentiation: cˢ_io > 0 доказан на равных ресурсах (контраст двойников
устраняет конфаунд упаковки), базовая c⁰_io отделена от надбавки, и показано, что
на дисковой оси **только** SensitivityScore защищает чувствительную задачу, тогда
как load-aware trimaran ей вредит.

Следующий шаг: сетевой вывод воркера (стрим на sink) для cˢ_net — каркас готов,
ждёт smoke-проверки связи «шторм → торможение стрима» на свободном кластере.

## Научная новизна и типовые вопросы

Новизна: (1) вектор чувствительности S отдельной задачи (Z = {G, R, S}) на
неквотируемых осях — против ресурсной модели R и одномерных load-aware
метрик; (2) сопряжённый вектор давления узла из штатных счётчиков
(cgroup-PMU, PSI) без инструментирования приложений; (3) двухкомпонентная
эмпирически калибруемая цена оси — базовая составляющая установлена
экспериментально и не выражается принятыми мультипликативными моделями;
(4) методика: разделение метрик решения и исхода, load-aware контроль,
предрегистрация критериев.

**«Почему выигрыш всего 1–3 %?»** (1) Выигрыш ограничен потолком стенда —
ценой ошибок наивного размещения: потолок default-плеча 2.2 с (11 задач ×
~12 с / 60), факт SS +2.7 с — совпадает с оценкой потолка; малый процент —
свойство стенда с квотами, мерить надо долей потолка и решениями.
(2) Хвост: p95 — 86 с (SS) против 95/97; максимум 88 против 103. Штраф
попавшей на токсичный узел задачи +15–25 % — метод защищает хвост.
(3) Процент растёт с ценой стенда: bare-metal включит кэш/NUMA
(литературные 10–40 %) при уже показанном ~идеальном качестве решений.
(4) 3 % среднего = 3 % ресурсов кластера заменой одного компонента.

**«Зачем оси, если Score — одно число?»** На выходе Score и есть одно
число на узел; оси — базис, без которого его нельзя ни построить, ни
перенести: (1) задачи с разной чувствительностью требуют разных чисел от
одного узла — единая скалярная «загруженность» и есть trimaran с его
35/60 на токсичном узле; (2) веса при осях определяются по вкладу оси в
замедление — это регрессия по измеренным осям (calibrate_axis_costs.py),
одно слитое число некалибруемо; (3) cˢ_io ≈ 0.63 > 0 (серия io-sensitivity)
даёт разным профилям разные числа от одного узла: чувствительная задача
уводится от диск-давления, нечувствительная — нет; при cˢ = 0 Score свёлся
бы к одному числу на узел, но вычисленному из осей со стендовыми ценами.

## Прогноз для прод-стенда (зарегистрирован 15.07.2026; стенд — 3× Dell R760, 2× Xeon Platinum 8462Y+ Sapphire Rapids, 384 ГБ, bare-metal)

1. Цены осей перевернутся: диск → <0.05 на NVMe (при сетевом хранилище
   останется); кэш — впервые β > 0, порядок +5–15 % (L3 у Xeon socket-wide,
   не CCX-группы как у EPYC — узловой датчик видит контраст L3 сокета целиком);
   NUMA-ось включится — 2 сокета / 2 NUMA-домена на узел (+10–30 % для MT при
   remote); сетевой референс вырастет на порядок.
2. Метрики решения переносимы; на больших узлах загрузка CPU ещё слабее
   отражает давление на диск/кэш/сеть.
3. Шум упадёт до CV 1–3 % — мощность выше при тех же 10 повторениях.
4. Скученность станет значимой (γ вырастет) — ожидается γ-слагаемое или
   прогноз давления через G.
5. Развилка: если β ≈ 0 и на полных ядрах — ценность S-вектора смещается
   к векторной детекции давления (главная проверка H1 на проде).

Калибровки (NET/LLC-референсы, цены осей) пересчитываются процедурой
Этапа 0 — переносится методика, не константы.
