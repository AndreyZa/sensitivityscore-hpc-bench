# Сводка результатов STAGE: четыре сценария фоновой нагрузки

**Главное.** Пройдены четыре серии (диск, сеть, кэш, смешанная — 3 шторма
одновременно), по 10 повторений × 3 планировщика. Механизм SensitivityScore
подтверждён по метрикам решения во всех четырёх сериях (ошибка размещения
и попадания на перегруженный узел — лучшие, вплоть до 0/60 в смешанной);
по времени выполнения выигрыш там, где ось дорогая (диск: δ = −1.0), и
проигрыш там, где дешёвая. Калибровка по собранным данным вскрыла причину —
скорингу не хватало *цены оси* — и дала уточнение модели: вклад оси
`давление·(α + β·чувствительность)`. Исправленный скоринг реализован и
проверен отдельной серией: **все три заранее объявленных критерия
выполнены** — проигрыш кэш-профиля устранён (δ с +0.95 до −0.06), выигрыши
сохранены, среднее время лучшее из трёх планировщиков (см. секцию
«Проверка исправленного скоринга»).

Стенд: Timeweb managed k8s (k0s), измерительные узлы + системный узел под
компонентами (роль `ss-system`). Серии кэша и смешанная — на PMU-тройке
w8/w9/w10 (эталонные времена однородны: 68–74 с, разброс ±2–3 %); серии
диска и сети — на более раннем составе из 6 узлов, включая аппаратно
~2× медленный w7 (неоднородность поглощена пер-узловыми эталонами —
замедление считается к эталону *того же* узла). Планировщики: default
(kube-scheduler), SensitivityScore (вектор чувствительности задачи
S = (LLC, NUMA, Net, IO) × давление узла), trimaran
(`LoadVariationRiskBalancing` — учёт утилизации CPU/памяти узла через
metrics-server, без модели интерференции). Статистика: Манн-Уитни по
повторениям, поправка Холма в рамках (сценарий, метрика), размер эффекта
Cliff's δ; 10 повторений на плечо. Полные отчёты:
`analysis/report-stage*/summary.md` — в репозитории не хранятся,
генерируются из данных; исходные данные всех серий (включая архивные)
загружены в ClickHouse на ПК-агрегаторе с провенансом (stand, run_label),
отчёт воспроизводится одной командой: `make ch-tunnel && make ch-analyze
STAND=stage RUN_LABEL=<серия>` (источники parquet/CH взаимозаменяемы,
сверено побайтово).

## Статус гипотез (формулировки — Программа экспериментов §6)

- **H1** (SensitivityScore лучше default по времени и стабильности при
  co-location) — **подтверждена частично, с уточнением модели**. По метрикам
  решения (ошибка размещения, попадания на перегруженный узел) — подтверждена
  во всех четырёх сценариях (диск: p_holm ≈ 4·10⁻⁴, δ до −1.0). По времени
  выполнения — подтверждена на дисковой оси (δ = −1.0), направленно на
  сетевой; **опровергнута в исходной формулировке скоринга** на кэш-оси и в
  смешанном сценарии — причина установлена (нет цены оси, см. калибровку),
  модель уточнена, и проверочная серия закрыла все три заранее объявленных
  критерия: с калиброванными ценами осей систематических проигрышей по
  времени не осталось (последняя секция).
- **H1-trimaran** (выигрыш даёт вектор чувствительности, а не «любой учёт
  загрузки») — **подтверждена**: trimaran проигрывает SensitivityScore по
  ошибке размещения во всех сериях; в смешанной ошибается систематически
  (35/60 задач на дисковый узел, «свободный» по CPU-метрике), диск-профиль —
  p_holm = 0.002, δ = −0.94 по времени. Слепота одномерной метрики к
  не-CPU-осям показана прямо.
- **H2 (KubeVirt), H3 (Slurm), H4 (Slinky)** — **не проверялись**: требуют
  конфигураций B/C/D прод-стенда (харнесс и манифесты готовы, см.
  Технический план §4 и дорожную карту Программы §9).

Побочные методические результаты, полученные по ходу (войдут в описание
методики Главы 3): инверсия отношения промахов кэша под потоковой нагрузкой
(датчик переведён на промахи/сек с калибровкой стенда); суточный дрейф
облачных vCPU +13–23 % (правило «эталоны — в ту же сессию, что серия»);
пер-узловые эталоны как защита от скрытой аппаратной неоднородности
(«одинаковые» облачные узлы бывают ~2× разной скорости).

## Смешанный сценарий: сопоставление вместо избегания

Постановка: чистых узлов нет — на каждом узле свой шторм (w8 — кэш,
w9 — диск, w10 — сеть); жертвы трёх профилей (по 2 кэш-, диск- и
сеть-чувствительных) приходят одним пуассоновским потоком. Выиграть можно
только сопоставлением: каждую задачу — на узел чужой оси.

Результат по решениям: SensitivityScore **ни разу из 60** не поставил задачу
на шторм её собственной оси (default — 21, trimaran — 22, причём trimaran
свёз 35/60 задач на дисковый узел: по CPU-метрике он выглядит свободным).

Результат по времени, по профилям (SS vs default / vs trimaran):

- диск-чувствительные: 83.2 с vs 85.4 / vs 91.0 с — против trimaran значимо
  (p_holm=0.002, δ −0.94), против default направленно лучше;
- сеть-чувствительные: 80.9 с vs 87.0 / 88.0 с — δ −0.57…−0.65, после Холма н/з;
- кэш-чувствительные: 91.3 с vs 78.7 / 92.2 с — **значимо хуже default**
  (p_holm=0.002, δ +0.95).

Причина видна в эмпирической матрице токсичности (время выполнения
профиль × узел по всем плечам):

| профиль | w8 (кэш-шторм) | w9 (диск-шторм) | w10 (сеть-шторм) |
|---|---|---|---|
| кэш-чувствительный | **77.9 с** | 96.2 с | 80.6 с |
| диск-чувствительный | 82.4 с | **97.0 с** | 85.4 с |
| сеть-чувствительный | 78.5 с | 94.6 с | 80.9 с |

Дисковый шторм токсичен **для всех** (+15–18 с любому профилю — iowait и
вытеснение страничного кэша бьют по узлу целиком), кэш-шторм при CPU-квотах
жертв 500m почти безвреден (узел с ним — самый быстрый даже для
кэш-чувствительных задач). SensitivityScore, не зная цены осей, увёл
кэш-чувствительные задачи с «их» узла w8 (эмпирически лучшего) на w9
(эмпирически худший для всех) — отсюда проигрыш default именно на этом
профиле при идеальной формальной метрике решения.

## Калибровка цен осей (по собранным данным, без новых прогонов)

Скрипт `analysis/calibrate_axis_costs.py` фитит по строкам LLC- и смешанной
серий (360 задач) модель «замедление − 1 ≈ γ·соседи + Σ давление_оси ×
(α_оси + β_оси·чувствительность задачи)» с посерийной константой
(суточный дрейф облачных vCPU между часом эталонов и часом серии).
Неотрицательный МНК; R² по ячейкам (профиль × узел) = 0.88.

| ось | α — базовая цена (платят все) | β — надбавка чувствительной задачи |
|---|---|---|
| диск (IO) | **+0.24** замедления при давлении 1.0 | 0 |
| кэш (LLC) | 0 | 0 |
| сеть (Net) | +0.02 | 0 |

Цена соседства γ = +0.026 на одну одновременно работающую соседнюю задачу;
дрейф сессии +0.13…+0.23 (ночные эталоны против дневной серии) — крупнейший
член модели, что количественно подтверждает правило «эталоны снимать в ту же
сессию, что и серию».

Три следствия:

1. **На этом стенде дорога только дисковая ось, и её цена базовая, а не
   чувствительностная** (β = 0 по всем осям): при CPU-квотах 500m кэш-шторм
   не замедляет даже кэш-чувствительную задачу, дисковый бьёт всех
   одинаково. Попутная находка: сетевой генератор (iperf3, 800 Мбит/с)
   выгоняет узловой llc_misses/s в насыщение — в смешанной серии «кэш-давление»
   узлов было 1.0 / 1.0 / 0.27, то есть контраста по кэшу между двумя
   узлами из трёх не было и по самой метрике.
2. **Веса как механизм калибровки недостаточны.** Проверка на матрице
   токсичности: текущие единичные веса выбирают эмпирически лучший узел для
   1 профиля из 3; калиброванные веса (β·s-скоринг как в плагине сейчас) —
   тоже 1 из 3, потому что задача с чувствительностью «low» к оси не платит
   за неё ничего при любом весе, а платить надо (α > 0). Расширенный скоринг
   `Σ давление·(α + β·чувствительность)` выбирает лучший узел для **3 из 3**.
3. **Различение задач на этом стенде не проявилось**: все три профиля-жертвы
   при квотах 500m фактически compute-bound (дисковые бурсты малы, сеть не
   генерируется, кэш зажат квотой) — измеренные цены не зависят от
   деклараций. Стенд валидирует детекцию давления и качество размещения;
   проверка пер-задачного различения (β > 0) требует стенда, где задачи
   реально упираются в свои оси — bare-metal с полноядерными квотами и/или
   утяжелённые IO/Net-профили.

## Выводы для дальнейшей работы

1. **Механизм подтверждён**: во всех сценариях SensitivityScore принимает
   лучшие решения по формальной метрике (ошибка размещения, попадания на
   токсичный узел), а там, где ось действительно дорогая, это конвертируется
   в значимый выигрыш времени с максимальным размером эффекта (диск: δ = −1.0).
2. **Ограничение модели — нет цены оси.** Вектор чувствительности ранжирует
   узлы по осям, но веса статичны и безразмерны: единица давления кэша
   считается столь же вредной, что и единица давления диска, хотя эмпирически
   они различаются на порядок. Два проявления: скученность в LLC-сценарии
   (избегание дешёвого шторма ценой взаимной интерференции на чистых узлах)
   и уход кэш-задач на дисковый узел в смешанном. Калибровка по собранным
   данным (см. секцию выше) показала, что перевзвешивания осей недостаточно:
   нужна базовая, не зависящая от чувствительности, цена оси в скоринге —
   `Σ давление·(α + β·чувствительность)` — плюс слагаемое цены соседства (γ).
3. **Одномерная CPU-метрика (trimaran) в смешанной нагрузке ошибается
   систематически**: дисковый узел выглядит свободным по CPU и собирает
   больше половины задач, включая диск-чувствительные (10/20 — на свой
   токсичный узел). Это основной аргумент в пользу векторного давления.

## Проверка исправленного скоринга — все критерии выполнены (15.07.2026)

Уточнение модели реализовано: вклад оси в скоре планировщика теперь
`давление·(α + β·чувствительность)` — базовая цена α тарифицирует давление
оси любой задаче, надбавка β — чувствительной (формат весов
`{"base": …, "sensitivity": …}` в ConfigMap; старый формат эквивалентен
β-части). Повтор смешанной серии: калиброванные цены (диск 1.0, сеть 0.09,
кэш 0), свежие эталоны той же сессией, поток задач с теми же моментами
прибытия, что в исходной серии; 180/180 задач, ошибок 0.

Критерии, объявленные до получения данных, против факта:

| # | Критерий | Было (равные веса) | Стало (калибровка) | Вердикт |
|---|---|---|---|---|
| 1 | кэш-профиль не проигрывает default (δ ≤ 0) | 91.3 vs 78.7 с, δ = **+0.95** | 77.7 vs 78.0 с, δ = **−0.06** | ✓ |
| 2 | выигрыши диска и сети сохранены | — | диск vs trimaran p_holm = 0.032, δ = −0.73; сеть δ = −0.56…−0.58; по ошибке размещения vs default: диск p_holm = 0.002 (δ = −0.90), сеть p_holm = 0.035 (δ = −0.64) | ✓ |
| 3 | среднее время SS ≤ default | 85.2 > 83.7 с | **78.8 ≤ 81.5 с** (trimaran 85.2) | ✓ |

Ключевые факты серии: SensitivityScore не разместил на дисковый узел
**ни одной из 60 задач** (default — 11, trimaran — 35); средняя ошибка
размещения 0.019 против 0.197 (default) и 0.546 (trimaran); кэш-профиль
против trimaran — p_holm = 0.006, δ = −0.88. Контроль дрейфа: плечи, не
зависящие от весов, ускорились между сериями на ~2.6 % (default
83.7 → 81.5 с) — сдвиг SensitivityScore (85.2 → 78.8 с) кратно больше
дрейфа, эффект принадлежит скорингу.

Показательное расцепление: номинальный счётчик «на узел своей
(декларированной) оси» у калиброванного планировщика вырос до 21/60 —
при лучшем времени и почти нулевой ошибке размещения. Планировщик,
знающий цены осей, занимает узлы дешёвых осей намеренно; номинальное
сопоставление перестало быть показателем качества (страница состояния
помечает его как справочный).

Честная оговорка: замедление к эталонам той же сессии вышло 0.92–0.96
(< 1) — внутрисессионный дрейф за 3.5 часа серии сохраняется даже ночью;
на сравнения плеч не влияет (плечи чередуются внутри каждого повторения).

Конфигурация серии: `harness/config-stage-mixed-calib.yaml`; отчёт —
`analysis/report-stage-mixed-calib`; данные — ClickHouse
(stand=stage, run_label=stage-mixed-calib).

## Научная новизна и ответ на вопрос «почему выигрыш по времени 1–3 %»

**Новизна метода** (кратко, для обсуждения формулировок):

1. Планирование по **пер-задачному вектору чувствительности** к
   интерференции (S в модели Z = {G, R, S}) на осях разделяемых ресурсов,
   не поддающихся квотированию (кэш, память NUMA, сеть, диск), — там, где
   существующие системы видят только ресурсные требования R, а load-aware
   подходы — одну метрику утилизации.
2. **Сопряжённый вектор давления узла** из штатных счётчиков (cgroup-PMU,
   PSI ядра) без инструментирования приложений, с пер-стендовой
   калибровкой порогов насыщения.
3. **Двухкомпонентная цена оси** — базовая (платит любая задача узла) +
   чувствительностная — с эмпирической калибровкой по измеренному
   замедлению. Установлено экспериментально, что базовая составляющая
   существует и может доминировать (дисковая ось: +24 % всем задачам);
   принятые в литературе мультипликативные модели интерференции этот
   вклад не выражают в принципе.
4. **Методика проверки**: разделение метрик решения и исхода (ошибка
   размещения по зафиксированному контексту решения — работает и для
   планировщиков вне измерительного контура), контрольное load-aware
   плечо, отделяющее учёт интерференции от учёта загрузки, критерии,
   объявляемые до получения данных.

**Про 1–3 % по времени** — четыре аргумента по нарастающей:

1. **Выигрыш ограничен потолком стенда, и потолок снят полностью.**
   Планировщик не ускоряет задачи — он не даёт им замедлиться; его
   максимум — цена ошибок наивного размещения. Арифметика смешанной
   серии: штраф дискового узла ~12 с/задачу × 11 задач default из 60 =
   потолок **2.2 с (2.7 %)**. SensitivityScore выиграл 2.7 с — весь
   потолок. У trimaran потолок 6.9 с — разрыв с SS 6.4 с, тоже почти
   весь. Малый процент — свойство стенда (CPU-квоты 500m глушат оси),
   а не метода; мерить метод надо долей снятого потолка (~100 %) и
   метриками решения (0/60 на дисковый узел, ошибка 0.019 против 0.197).
2. **Хвост важнее среднего.** Для задачи, попавшей на токсичный узел,
   штраф индивидуален: +15–25 %. Смешанная серия: p95 времени —
   SS **86 с** против 95 (default) и 97 (trimaran); максимум — **88**
   против 103 и 101. Метод — защита от хвостовых размещений; в HPC
   предсказуемость и p95 (цепочки заданий, дедлайны) ценнее 3 % среднего.
3. **Процент растёт с ценой интерференции стенда.** На bare-metal с
   полноядерными квотами включаются кэш и NUMA (литературные диапазоны
   деградации 10–40 %), плотность co-location выше. Качество решений уже
   показано близким к идеалу — большая цена осей конвертируется тем же
   механизмом в больший процент (см. зарегистрированный прогноз ниже).
4. **Системный эффект.** 3 % среднего времени на кластере — это 3 %
   ресурсов кластера, полученные заменой одного компонента без изменения
   приложений; плюс ноль катастрофических размещений (35/60 задач
   trimaran на токсичном узле — цена альтернативы).

## Прогноз для прод-стенда (зарегистрирован до развёртывания, 15.07.2026)

Ожидаемый прод: bare-metal, крупные узлы на AMD EPYC. Проверяемые
предсказания — сверить с фактом после ввода стенда:

1. **Цены осей перевернутся.** Диск: базовая цена (0.24 на STAGE) на
   локальных NVMe упадёт к ~0 (<0.05); при сетевом хранилище — останется.
   Кэш: впервые cˢ > 0 (полноядерные квоты, MT-Geant4 с HP-таблицами),
   порядок +5–15 % при полном шторме — с оговоркой, что L3 у EPYC
   пер-CCX, узловой датчик разбавляет контраст (кандидат на уточнение:
   per-CCD агрегация). NUMA: ось включается впервые (NPS>1 / два сокета),
   ожидаемый класс эффектов +10–30 % для MT-профиля при remote-размещении.
   Сеть: референс вырастет на порядок, генераторы масштабировать.
2. **Метрики решения реплицируются**; слепота CPU-метрики (trimaran)
   станет резче — шторм одной подсистемы почти не двигает утилизацию
   большого узла.
3. **Шум упадёт**: CV эталонов ~1–3 % (на STAGE 3–7 %, плюс суточный
   дрейф до 23 %) — статистическая мощность выше при тех же 10
   повторениях.
4. **Скученность станет значимой**: больше со-размещений на узел → цена
   соседства γ (0.026 на STAGE) вырастет; ожидается потребность в
   γ-слагаемом скоринга или прогнозе давления через вектор G.
5. **Ключевая развилка**: если cˢ ≈ 0 и на полных ядрах — ценность
   S-вектора смещается с пер-задачного сопоставления на векторную
   детекцию давления (H1 в исходной формулировке слабеет). Оценка:
   маловероятно, но это главная проверка прода.

Штормы и все калибровки (NET/LLC-референсы, цены осей) пересчитываются
на месте процедурой Этапа 0 — переносится методика, не константы.
