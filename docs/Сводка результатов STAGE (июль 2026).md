# Сводка результатов STAGE (июль 2026)

**Итог.** Пять серий: диск, сеть, кэш, смешанная (три шторма одновременно),
смешанная с калиброванным скорингом; 3 планировщика × 10 повторений.
По метрикам решения SensitivityScore лучший во всех сериях. По времени —
выигрыш на дорогих осях; два проигрыша на дешёвых вскрыли ограничение
модели (нет цены оси). Калибровка дала уточнение — вклад оси =
давление·(α + β·чувствительность) — и отдельная серия закрыла все три
заранее объявленных критерия проверки.

Стенд: Timeweb managed k8s (k0s), системные компоненты на выделенном узле
(роль `ss-system`). Серии кэша и смешанные — на PMU-тройке w8/w9/w10
(эталоны однородны, 68–74 с ±2–3 %); диск и сеть — на раннем составе из
6 узлов, включая ~2× медленный w7 (поглощается эталонами на каждом узле).
Планировщики: default, SensitivityScore, trimaran
(`LoadVariationRiskBalancing` — утилизация CPU/памяти, без модели
интерференции). Статистика: Манн-Уитни по повторениям, поправка Холма,
Cliff's δ. Данные всех серий — в ClickHouse с провенансом; отчёт
воспроизводится: `make ch-tunnel && make ch-analyze STAND=stage
RUN_LABEL=<серия>`.

## Статус гипотез

- **H1** — по метрикам решения подтверждена во всех сериях; по времени
  подтверждена на дорогих осях, опровергнута в исходной форме скоринга на
  дешёвых; после уточнения модели проверочная серия систематических
  проигрышей не оставила. Окончательная проверка различения задач по чувствительности
  (β > 0) — прод.
- **H1-trimaran** — подтверждена: load-aware проигрывает по решениям
  везде; в смешанных сериях систематическая ошибка (до 35/60 задач на
  дисковый узел, «свободный» по CPU).
- **H2–H4** — не проверялись (нужны конфигурации B/C/D).

Методические результаты по пути: инверсия отношения промахов кэша под
потоковой нагрузкой (датчик переведён на промахи/сек с калибровкой);
суточный дрейф облачных vCPU +13–23 % → правило «эталоны в ту же сессию»;
эталоны на каждом узле против скрытой неоднородности (узлы бывают ~2× разными).

## Почему проигрывал исходный скоринг: цены осей

Матрица токсичности смешанной серии (время профиль × узел):

| профиль | w8 (кэш-шторм) | w9 (диск-шторм) | w10 (сеть-шторм) |
|---|---|---|---|
| кэш-чувствительный | **77.9 с** | 96.2 с | 80.6 с |
| диск-чувствительный | 82.4 с | **97.0 с** | 85.4 с |
| сеть-чувствительный | 78.5 с | 94.6 с | 80.9 с |

Дисковый шторм замедляет все профили (+15–18 с); кэш-шторм при квотах 500m практически не влияет.
Калибровка по 360 задачам (модель «замедление ≈ γ·соседи + Σ p·(α + β·s)»,
NNLS, посерийный интерсепт; R² по ячейкам 0.88):

| ось | α — базовая цена (платят все) | β — надбавка чувствительной |
|---|---|---|
| диск | **+0.24** | 0 |
| кэш | 0 | 0 |
| сеть | +0.02 | 0 |

γ (сосед) = +0.026; дрейф сессии +0.13…+0.23 — крупнейший член. Следствия:
(1) цена оси базовая — скоринг Σ w·s·p её не выражает (задача с s = 0 не
платит при любых весах), перевзвешивание не лечит; (2) на матрице
токсичности исходный скоринг выбирает лучший узел для 1 профиля из 3,
расширенный p·(α + β·s) — для 3 из 3; (3) при квотах 500m задачи
compute-bound — различение по декларациям (β > 0) на этом стенде
непроверяемо.

## Проверка исправленного скоринга — все критерии выполнены (15.07)

эквивалентен β-части). Серия: те же шторма и поток (детерминированные
сиды), эталоны той же сессией, 180/180 задач без ошибок.

Контроль дрейфа: независимые от весов плечи ускорились на ~2.6 %, сдвиг

своей оси» при этом 21/60 — при лучшем времени: планировщик, знающий
цены, занимает дешёвые оси намеренно. Оговорка: замедления к эталонам
той же сессии 0.92–0.96 — внутрисессионный дрейф остаётся; на сравнения
плеч не влияет (плечи чередуются в каждом повторении).

## Научная новизна и типовые вопросы

Новизна: (1) вектор чувствительности S отдельной задачи (Z = {G, R, S}) на
неквотируемых осях — против ресурсной модели R и одномерных load-aware
метрик; (2) сопряжённый вектор давления узла из штатных счётчиков
(cgroup-PMU, PSI) без инструментирования приложений; (3) двухкомпонентная
эмпирически калибруемая цена оси — базовая составляющая установлена
экспериментально и не выражается принятыми мультипликативными моделями;
(4) методика: разделение метрик решения и исхода, load-aware контроль,
предрегистрация критериев.

**«Почему выигрыш всего 1–3 %?»** (1) Выигрыш ограничен потолком стенда —
ценой ошибок наивного размещения: потолок default-плеча 2.2 с (11 задач ×
~12 с / 60), факт SS +2.7 с — совпадает с оценкой потолка; малый процент —
свойство стенда с квотами, мерить надо долей потолка и решениями.
(2) Хвост: p95 — 86 с (SS) против 95/97; максимум 88 против 103. Штраф
попавшей на токсичный узел задачи +15–25 % — метод защищает хвост.
(3) Процент растёт с ценой стенда: bare-metal включит кэш/NUMA
(литературные 10–40 %) при уже показанном ~идеальном качестве решений.
(4) 3 % среднего = 3 % ресурсов кластера заменой одного компонента.

**«Зачем оси, если Score — одно число?»** На выходе Score и есть одно
число на узел; оси — базис, без которого его нельзя ни построить, ни
перенести: (1) задачи с разной чувствительностью требуют разных чисел от
одного узла — единая скалярная «загруженность» и есть trimaran с его
35/60 на токсичном узле; (2) веса при осях определяются по вкладу оси в
замедление — это регрессия по измеренным осям (calibrate_axis_costs.py),
одно слитое число некалибруемо; (3) вырожденный случай β = 0 (STAGE)
действительно сводит Score к одному числу на узел — но вычисленному из
осей со стендовыми ценами, и переносимость обеспечивают именно оси.

## Прогноз для прод-стенда (зарегистрирован 15.07.2026, EPYC bare-metal)

1. Цены осей перевернутся: диск → <0.05 на NVMe (при сетевом хранилище
   останется); кэш — впервые β > 0, порядок +5–15 % (оговорка: L3 у EPYC
   на уровне CCX (группы ядер), узловой датчик разбавляет контраст); NUMA-ось включится
   (+10–30 % для MT при remote); сетевой референс вырастет на порядок.
2. Метрики решения переносимы; на больших узлах загрузка CPU ещё слабее
   отражает давление на диск/кэш/сеть.
3. Шум упадёт до CV 1–3 % — мощность выше при тех же 10 повторениях.
4. Скученность станет значимой (γ вырастет) — ожидается γ-слагаемое или
   прогноз давления через G.
5. Развилка: если β ≈ 0 и на полных ядрах — ценность S-вектора смещается
   к векторной детекции давления (главная проверка H1 на проде).

Калибровки (NET/LLC-референсы, цены осей) пересчитываются процедурой
Этапа 0 — переносится методика, не константы.
