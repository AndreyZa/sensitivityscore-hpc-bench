# Сводка результатов STAGE: четыре сценария фоновой нагрузки

Стенд: Timeweb managed k8s (k0s), три измерительных узла (w8/w9/w10 — PMU-тройка,
эталонные времена однородны: 68–74 с, разброс ±2–3 %), системный узел под
компонентами (роль `ss-system`). Планировщики: default (kube-scheduler),
SensitivityScore (вектор чувствительности задачи S = (LLC, NUMA, Net, IO) ×
давление узла), trimaran (TargetLoadPacking — одна CPU-метрика узла).
Статистика: Манн-Уитни по повторениям, поправка Холма в рамках
(сценарий, метрика), размер эффекта Cliff's δ; 10 повторений на плечо.
Полные отчёты: `analysis/report-stage*/summary.md` (генерируются из parquet,
в репозитории не хранятся).

| Диск (IO) | попаданий в шторм: ~0 % | 0.001 / 0.113 / 0.166 — SS лучше обоих, p≈4·10⁻⁴, δ −0.96…−1.0 | SS лучше обоих, δ = −1.0 |

| Смешанный (3 шторма) | на свой токсичный узел: **0/60** / 21/60 / 22/60 | 0.129 / 0.208 / 0.201 | среднее 85.2 / 83.7 / 90.4 с; по профилям — см. ниже |

Метрика решения (куда планировщик поставил задачу) устойчиво лучшая у
SensitivityScore во всех четырёх сценариях — механизм работает: пер-задачный
вектор чувствительности действительно уводит задачу от давления её оси.
Время выполнения выигрывается там, где ось дорогая (диск, сеть), и
проигрывается там, где избегание дешёвой оси загоняет задачи на дорогую
(кэш, смешанный — подробнее ниже).

## Смешанный сценарий: сопоставление вместо избегания

Постановка: чистых узлов нет — на каждом узле свой шторм (w8 — кэш,
w9 — диск, w10 — сеть); жертвы трёх профилей (по 2 кэш-, диск- и
сеть-чувствительных) приходят одним пуассоновским потоком. Выиграть можно
только сопоставлением: каждую задачу — на узел чужой оси.

Результат по решениям: SensitivityScore **ни разу из 60** не поставил задачу
на шторм её собственной оси (default — 21, trimaran — 22, причём trimaran
свёз 35/60 задач на дисковый узел: по CPU-метрике он выглядит свободным).

Результат по времени, по профилям (SS vs default / vs trimaran):

- диск-чувствительные: 83.2 с vs 85.4 / vs 91.0 с — против trimaran значимо
  (p_holm=0.002, δ −0.94), против default направленно лучше;
- сеть-чувствительные: 80.9 с vs 87.0 / 88.0 с — δ −0.57…−0.65, после Холма н/з;
- кэш-чувствительные: 91.3 с vs 78.7 / 92.2 с — **значимо хуже default**
  (p_holm=0.002, δ +0.95).

Причина видна в эмпирической матрице токсичности (время выполнения
профиль × узел по всем плечам):

| профиль | w8 (кэш-шторм) | w9 (диск-шторм) | w10 (сеть-шторм) |
|---|---|---|---|
| кэш-чувствительный | **77.9 с** | 96.2 с | 80.6 с |
| диск-чувствительный | 82.4 с | **97.0 с** | 85.4 с |
| сеть-чувствительный | 78.5 с | 94.6 с | 80.9 с |

Дисковый шторм токсичен **для всех** (+15–18 с любому профилю — iowait и
вытеснение страничного кэша бьют по узлу целиком), кэш-шторм при CPU-квотах
жертв 500m почти безвреден (узел с ним — самый быстрый даже для
кэш-чувствительных задач). SensitivityScore, не зная цены осей, увёл
кэш-чувствительные задачи с «их» узла w8 (эмпирически лучшего) на w9
(эмпирически худший для всех) — отсюда проигрыш default именно на этом
профиле при идеальной формальной метрике решения.

## Выводы для дальнейшей работы

1. **Механизм подтверждён**: во всех сценариях SensitivityScore принимает
   лучшие решения по формальной метрике (ошибка размещения, попадания на
   токсичный узел), а там, где ось действительно дорогая, это конвертируется
   в значимый выигрыш времени с максимальным размером эффекта (диск: δ = −1.0).
2. **Ограничение модели — нет цены оси.** Вектор чувствительности ранжирует
   узлы по осям, но веса статичны и безразмерны: единица давления кэша
   считается столь же вредной, что и единица давления диска, хотя эмпирически
   они различаются на порядок. Два проявления: скученность в LLC-сценарии
   (избегание дешёвого шторма ценой взаимной интерференции на чистых узлах)
   и уход кэш-задач на дисковый узел в смешанном. Направление: эмпирическая
   калибровка веса оси в единицах замедления (замедление на единицу давления,
   по данным эталонных прогонов + микрошторма), либо слагаемое предельной
   цены соседства в скоринге.
3. **Одномерная CPU-метрика (trimaran) в смешанной нагрузке ошибается
   систематически**: дисковый узел выглядит свободным по CPU и собирает
   больше половины задач, включая диск-чувствительные (10/20 — на свой
   токсичный узел). Это основной аргумент в пользу векторного давления.
