# Глоссарий

Термины эксперимента — в порядке «кто есть кто», затем метрики. В скобках —
имя колонки в данных (parquet/ClickHouse), где оно есть.

## Участники эксперимента

| Термин | Что это |
|---|---|
| **Ось** | Один разделяемый ресурс узла, не управляемый квотами CPU/памяти: кэш последнего уровня (LLC), диск (IO), сеть (Net), NUMA. По одной компоненте вектора на ось. |
| **Вектор чувствительности S** | Декларация задачи «насколько мне дорога каждая ось», числа 0..1. Задаётся аннотациями пода. |
| **Давление узла** | Измеренная загруженность оси узла, 0..1, безотносительно того, кто её создал. Меряет метрики-агент на каждом узле, планировщик читает из Redis. |
| **Генератор нагрузки («агрессор»)** | Служебный под (stress-ng или iperf3), создающий давление на одной оси одного узла. Не измеряется и в результаты не входит. |
| **Шторм** | Группа генераторов одной оси на одном узле (у нас — 2 шт.). «Дисковый шторм на w9» = узел w9 с давлением на ось диска. Сетевой шторм бывает двух видов: *локальный* (пара на одном узле — датчик давление видит, но обмен соседа не страдает, только для детекции) и *egress* (трафик уходит на другой узел, насыщает аплинк и реально замедляет обмен соседа ×5.2). |
| **Жертва** | Измеряемая задача Geant4, поступающая в кластер через испытуемый планировщик. Только строки жертв попадают в результаты. |
| **Профиль** | Вариант жертвы с заданной чувствительностью: `low-s` (низкая по всем осям), `high-s` (кэш), `high-s-io` (диск, вывод на критическом пути), `high-s-net` (сеть, в режиме `stream` — сетевой вывод на критическом пути). |
| **Задача-близнец** | Профиль, идентичный чувствительному по расчёту и ресурсам, но без вывода и с низкой заявленной чувствительностью (`io-insensitive`, `net-insensitive`). Единственное различие пары — чувствительность, поэтому разница их времени под одним штормом изолирует цену cˢ от упаковки по ресурсам. |
| **Плечо** | Один и тот же поток жертв под одним планировщиком: default, SensitivityScore или trimaran. Плечи одного повторения получают одинаковые моменты прибытия задач. |
| **Серия** | Полный прогон сценария: все плечи × все повторения, одной сессией с эталонами. |
| **Сессия** | Эталоны + серия, выполненные подряд: производительность облачных vCPU дрейфует в течение суток (+13–23 % между ночью и днём), сравнивать время можно только внутри сессии. |
| **Плацебо-серия** | Серия без генераторов нагрузки (отрицательный контроль): на чистом кластере различий между планировщиками быть не должно. |
| **Харнесс** (`harness/`) | Управляющая программа эксперимента: разворачивает генераторы, подаёт поток жертв через испытуемый планировщик, снимает метрики и пишет результаты. Запуск серии — одна команда (`make series SERIES=<имя>`). |

## Метрики и понятия анализа

| Термин | Что это |
|---|---|
| **Эталон** (`baseline`) | Соло-прогон профиля на пустом узле с принудительной привязкой — знаменатель замедления, свой для каждой пары (профиль, узел). |
| **Время выполнения** (`makespan_s`) | Время задачи от старта до завершения, секунды. |
| **Замедление** (`slowdown`) | Время задачи ÷ эталон того же профиля на том же узле той же сессии. 1.0 — как в одиночестве. |
| **Ошибка размещения** (`placement_regret`) | Насколько интерференция выбранного узла хуже лучшего доступного в момент решения, 0..1 (0 — выбран лучший узел, 1 — худший). Считается по снапшоту давления из Redis; от дрейфа стенда не зависит. |
| **Токсичность** | Цена узла со штормом для конкретного профиля: узел «токсичен» для задачи, если его шторм её ощутимо замедляет. *Номинальная* — по совпадению оси шторма с заявленной чувствительностью; *эмпирическая* — по матрице токсичности (измеренное время профиль × узел). На STAGE они расходятся: дисковый шторм замедляет все профили, кэшевый при квотах 500m — почти никого. |
| **Цена оси** | Замедление на единицу давления оси: базовая часть (платит любая задача узла) + надбавка чувствительной. Коэффициенты скоринга; калибруются регрессией по собранным сериям, свои для каждого стенда. |
| **Fingerprint** | Сверка заявленной чувствительности профиля с измеренной в эталонных прогонах — ловит расхождение декларации и фактического поведения. |
| **Дрейф стенда** | Медленное изменение производительности узлов (суточный цикл steal облачных vCPU). Контролируется правилом сессии и плечами, не зависящими от весов. |
| **Провенанс** (`stand`, `run_label`) | Метки происхождения записи в БД: какой стенд и какая серия её породили. По ним любой отчёт воспроизводится точно из ClickHouse. |

Как термины связаны в одном предложении: в серии на узлах разворачиваются
**шторма**, планировщик каждого **плеча** размещает поток **жертв**, по каждой
жертве записываются **время**, **замедление** к **эталону** и **ошибка
размещения**; хороший планировщик избегает **токсичных** для задачи узлов,
зная **цены осей** и **давление** узлов.
