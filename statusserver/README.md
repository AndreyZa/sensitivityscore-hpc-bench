# statusserver — страница прогресса прогона

HTTP-страница, показывающая, что происходит с идущей серией экспериментов:
фаза, процент и ETA, план прогона, доля задач, уехавших на перегруженный узел,
живые Job'ы и агрессоры, матрица эталонов, хвост лога. Плюс `/json` для
скриптов.

**Только чтение.** Ни один путь не открывается на запись, монтирования и RBAC
read-only. Страницу держат открытой во время многочасовой серии, и возможности
испортить прогон у неё быть не должно.

## Запуск

Обычно запускать вручную не нужно: страницу поднимает сам лаунчер серии.
`scripts/run-series.sh` делает это при старте — и через `make series`, и при
ручном запуске `harness/run-stage-<серия>.sh` (вызов вшит в их шапку).
Повторный вызов идемпотентен: если нужная страница уже отвечает, контейнер не
трогается.

Вручную — когда надо посмотреть на уже законченную серию:

```bash
make status-page SERIES=net-diff        # поднять
make series-status SERIES=net-diff      # состояние страницы + серии
make status-page-down                   # погасить
```

Под капотом — `statusserver/docker-compose.yaml`:

```bash
SERIES=net-diff docker compose -f statusserver/docker-compose.yaml up -d --build
docker compose -f statusserver/docker-compose.yaml logs -f
docker compose -f statusserver/docker-compose.yaml down
```

`--build` обязателен: без него поднимется ранее собранный `statusserver:local`
и правки страницы в контейнер не попадут.

Переменные: `SERIES` (какую серию смотреть), `STATUS_PORT` (порт на хосте,
8787), `STAND` (подпись в шапке), `KUBECONFIG` (доступ к стенду),
`RESULTS`/`BASELINES` (пути к parquet — их подставляет `run-series.sh` из
секции `output` конфига серии).

`make series-stop` страницу **не** гасит — она только читает и полезна после
прогона. Гасить явно: `make status-page-down`.

### Почему контейнер, а не хостовый python

Не ради удобства. На хосте страница умирала с SIGSEGV (код 139, пустой лог) —
и в контейнере продолжала умирать так же, пока причину не нашли: дефолтный
memory pool Arrow роняет процесс на `pd.read_parquet`, вызванном из
тред-обработчика `ThreadingHTTPServer`. Воспроизводится 60 последовательными
запросами к странице на реальном parquet серии; трейсбек виден только под
`PYTHONFAULTHANDLER=1`. Лечится `ARROW_DEFAULT_MEMORY_POOL=system` — переменная
стоит в `Dockerfile` и продублирована в кластерном манифесте.

Первоначальный диагноз («виноват python 3.14») был неверным: интерпретатор ни
при чём, подпись падения одна и та же на 3.14 и 3.12. Контейнер нужен не как
лекарство, а как способ зафиксировать окружение вместе с этой переменной.

## В кластере

```bash
make image-statusserver && docker push andreyza/statusserver:dev
make statusserver-deploy      # ss-system, PVC харнесса read-only
make statusserver-open        # port-forward на localhost:8787
```

Образ для кластера собирается под `linux/amd64`; локальный compose собирает
**свой** нативный `statusserver:local` — на Apple Silicon amd64-образ идёт под
эмуляцией и молча выходит. Это разные образы, и локально пушить ничего не надо.

Кластерный манифест пока неполон — см. список ограничений в шапке
`k8s/statusserver/statusserver.yaml` (нодо-локальный RWO-том против пиннинга к
ss-system; лог и конфиг серии в кластер никто не кладёт; под в bench-namespace
против требования preflight «namespace пуст»). Разбирать при вводе прод-стенда.

## Эндпойнты

| путь | что делает |
|---|---|
| `/` | HTML-страница, авто-обновление настраивается в шапке |
| `/json` | тот же снимок в JSON |
| `/healthz` | дешёвая проверка живости, ничего не читает |
| `/report/<файл>` | файлы из каталога `--report` (png/csv/md), без выхода за его пределы |

`/healthz` отделён намеренно: `/` собирает весь снимок (лог, оба parquet, до
трёх вызовов `kubectl`), и на холодном кэше топологии это дольше таймаута
проверки — healthcheck метил исправную страницу как unhealthy, а в кластере
readinessProbe выводила под из Endpoints.

## Когда цифрам верить нельзя

Страница показывает плашки вместо молчаливой лжи:

- **«Конфиг серии не прочитан»** — не нашёлся или не разобрался конфиг
  (частый случай — опечатка в `SERIES`). Прогресс-бар и план недоступны.
- **«Список измерительных узлов не получен»** — `kubectl` не ответил, объём
  эталонов неизвестен. Можно задать явно: `baseline.nodes` в конфиге серии.
- **«Прогон помечен завершённым, но не хватает N строк»** — маркер
  `=== PRESSURE DONE ===` печатается скриптом серии безусловно, чем бы харнесс
  ни кончился. Раньше серия, упавшая на первом плече, показывала «завершён ✓
  100%».

## Модули

`server` — сборка снимка (`collect`) и HTTP; `data` — parquet/csv → сводки;
`progress` — лог → фаза/проценты/ETA/план; `cluster` — снимки через `kubectl`;
`render` — HTML; `labels` — подписи и словоформы; `selfcheck` — проверка, что
модули импортируются и страница рендерится на синтетике.

Терминология на странице рассчитана на читателя со стороны (научный
руководитель), а не на жаргон проекта: «планировщик», «повторение»,
«эталонные прогоны», «перегруженный узел».
